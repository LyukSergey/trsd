# L31 - JPA Relationships & Transactions

## Що таке зв'язки між сутностями?

В реляцiйних базах даних таблицi пов'язанi мiж собою через зовнiшнi ключi (Foreign Keys). JPA дозволяє описати цi зв'язки прямо в Java-класах за допомогою анотацiй. Hibernate пiд капотом генерує потрiбний SQL.

Є три основнi типи зв'язкiв:

| Тип зв'язку | Приклад з життя | Приклад з проекту |
|-------------|-----------------|-------------------|
| OneToMany / ManyToOne | Один клас — багато студентiв | Bank -> Users |
| OneToOne | Одна людина — один паспорт | User <-> UserProfile |
| ManyToMany | Студент вiдвiдує багато курсiв, курс має багато студентiв | User <-> Role |

## 1. OneToMany / ManyToOne

**Суть:** один банк має багато користувачiв. Кожен користувач належить одному банку.

### Як це виглядає в БД

```
bank                    users
+----+-------------+   +----+--------+----------+---------+
| id | name        |   | id | name   | surname  | bank_id |
+----+-------------+   +----+--------+----------+---------+
| 1  | Приват Банк |   | 1  | Iван   | Франко   | 1       |
|    |             |   | 2  | Тарас  | Шевченко | 1       |
|    |             |   | 3  | Леся   | Українка | 1       |
+----+-------------+   +----+--------+----------+---------+
```

`bank_id` — це зовнiшнiй ключ (FK), який посилається на `bank.id`.

### Сторона "One" — Bank

```java
@Entity
@Table(name = "bank")
public class Bank {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    @OneToMany(mappedBy = "bank", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<User> users = new ArrayList<>();
}
```

- `@OneToMany` — один банк має багато юзерiв
- `mappedBy = "bank"` — зв'язок керується полем `bank` в класi `User` (FK знаходиться там)
- `cascade = CascadeType.ALL` — операцiї (save, delete) каскадно поширюються на юзерiв
- `orphanRemoval = true` — якщо юзера видалити зi списку, його видалять i з БД

### Сторона "Many" — User

```java
@Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
    private String surname;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "bank_id")
    private Bank bank;
}
```

- `@ManyToOne` — багато юзерiв належать одному банку
- `@JoinColumn(name = "bank_id")` — вказує колонку-FK в таблицi `users`
- `fetch = FetchType.LAZY` — банк завантажується тiльки коли ми до нього звернемось (не одразу)

### Liquibase-мiграцiя

```xml
<createTable tableName="users">
    <column name="id" type="BIGINT" autoIncrement="true">
        <constraints primaryKey="true"/>
    </column>
    <column name="name" type="VARCHAR(255)"/>
    <column name="surname" type="VARCHAR(255)"/>
    <column name="bank_id" type="BIGINT"/>
</createTable>

<addForeignKeyConstraint baseTableName="users" baseColumnNames="bank_id"
    constraintName="fk_users_bank"
    referencedTableName="bank" referencedColumnNames="id"/>
```

### Iнтеграцiйний тест

```java
@Test
void testOneToMany_BankHasManyUsers() {
    Bank bank = new Bank();
    bank.setName("Test Bank");
    bank.setTotalAmount(1000000);
    bank = bankRepository.save(bank);

    User user1 = new User();
    user1.setName("Тарас");
    user1.setSurname("Шевченко");
    user1.setBank(bank);    // <-- встановлюємо зв'язок

    User user2 = new User();
    user2.setName("Леся");
    user2.setSurname("Українка");
    user2.setBank(bank);

    userRepository.save(user1);
    userRepository.save(user2);

    // Перевiряємо: банк має 2 юзерiв
    var users = userRepository.findAllByBankId(bank.getId());
    assertEquals(2, users.size());

    // Зворотнiй зв'язок: юзер -> банк
    Optional<Bank> userBank = bankRepository.findByUsersId(user1.getId());
    assertEquals("Test Bank", userBank.get().getName());
}
```

### Ключовий момент

**Хто "володiє" зв'язком?** Сторона з `@JoinColumn` — тобто `User`. Саме в таблицi `users` лежить колонка `bank_id`. Тому при збереженнi потрiбно встановлювати `user.setBank(bank)`, а не додавати юзера в `bank.getUsers()`.

## 2. OneToOne

**Суть:** кожен користувач має рiвно один профiль. Профiль належить рiвно одному користувачу.

### Як це виглядає в БД

```
users                        user_profile
+----+-------+----------+   +----+--------------------+---------+
| id | name  | surname  |   | id | email              | user_id |
+----+-------+----------+   +----+--------------------+---------+
| 1  | Iван  | Франко   |   | 1  | ivan.franko@ex.com | 1       |
+----+-------+----------+   +----+--------------------+---------+
```

`user_id` в `user_profile` — FK з UNIQUE constraint (гарантує що один юзер = один профiль).

### Сторона-власник — UserProfile

```java
@Entity
@Table(name = "user_profile")
public class UserProfile {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String email;
    private String phone;

    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", unique = true)
    private User user;
}
```

- `@JoinColumn(name = "user_id", unique = true)` — FK-колонка з обмеженням унiкальностi

### Зворотня сторона — User

```java
@Entity
@Table(name = "users")
public class User {
    // ... iншi поля

    @OneToOne(mappedBy = "user", cascade = CascadeType.ALL, orphanRemoval = true)
    private UserProfile profile;
}
```

- `mappedBy = "user"` — зв'язком керує поле `user` в `UserProfile`
- `cascade = CascadeType.ALL` — при збереженнi юзера автоматично зберiгається i профiль

### Iнтеграцiйний тест

```java
@Test
void testOneToOne_UserHasOneProfile() {
    User user = new User();
    user.setName("Iван");
    user.setSurname("Франко");
    user = userRepository.save(user);

    UserProfile profile = new UserProfile();
    profile.setEmail("franko@ukr.net");
    profile.setPhone("+380991234567");
    profile.setUser(user);              // <-- встановлюємо зв'язок
    userProfileRepository.save(profile);

    // Перевiряємо
    UserProfile found = userProfileRepository.findByUserId(user.getId()).orElseThrow();
    assertEquals("franko@ukr.net", found.getEmail());
    assertEquals(user.getId(), found.getUser().getId());
}
```

## 3. ManyToMany

**Суть:** користувач може мати багато ролей. Роль може належати багатьом користувачам.

### Як це виглядає в БД

Для ManyToMany потрiбна **промiжна таблиця** (join table):

```
users          user_role          role
+----+------+  +---------+--------+  +----+---------+
| id | name |  | user_id | role_id|  | id | name    |
+----+------+  +---------+--------+  +----+---------+
| 1  | Iван |  | 1       | 1      |  | 1  | ADMIN   |
| 3  | Леся |  | 1       | 2      |  | 2  | USER    |
+----+------+  | 3       | 2      |  | 3  | MANAGER |
               | 3       | 3      |  +----+---------+
               +---------+--------+
```

Iван має ролi ADMIN + USER. Леся має ролi USER + MANAGER.

### Сторона-власник — User

```java
@Entity
@Table(name = "users")
public class User {
    // ... iншi поля

    @ManyToMany
    @JoinTable(
            name = "user_role",                                    // iм'я промiжної таблицi
            joinColumns = @JoinColumn(name = "user_id"),           // FK на цю сутнiсть
            inverseJoinColumns = @JoinColumn(name = "role_id")     // FK на iншу сутнiсть
    )
    private Set<Role> roles = new HashSet<>();
}
```

### Зворотня сторона — Role

```java
@Entity
@Table(name = "role")
public class Role {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(unique = true)
    private String name;

    @ManyToMany(mappedBy = "roles")
    private Set<User> users = new HashSet<>();
}
```

- `mappedBy = "roles"` — зв'язком керує поле `roles` в `User`
- Використовуємо `Set`, а не `List` — для ManyToMany це правильнiше (уникаємо дублiкатiв i проблем з Hibernate)

### Liquibase-мiграцiя промiжної таблицi

```xml
<createTable tableName="user_role">
    <column name="user_id" type="BIGINT">
        <constraints nullable="false"/>
    </column>
    <column name="role_id" type="BIGINT">
        <constraints nullable="false"/>
    </column>
</createTable>
<addPrimaryKey tableName="user_role" columnNames="user_id, role_id"/>
<addForeignKeyConstraint baseTableName="user_role" baseColumnNames="user_id"
    constraintName="fk_user_role_user"
    referencedTableName="users" referencedColumnNames="id"/>
<addForeignKeyConstraint baseTableName="user_role" baseColumnNames="role_id"
    constraintName="fk_user_role_role"
    referencedTableName="role" referencedColumnNames="id"/>
```

Зверни увагу: **composite primary key** `(user_id, role_id)` — одна пара може iснувати тiльки один раз.

### Iнтеграцiйний тест

```java
@Test
void testManyToMany_UserHasManyRoles() {
    Role admin = new Role();
    admin.setName("ADMIN");
    admin = roleRepository.save(admin);

    Role manager = new Role();
    manager.setName("MANAGER");
    manager = roleRepository.save(manager);

    User user = new User();
    user.setName("Богдан");
    user.setSurname("Хмельницький");
    user.getRoles().add(admin);     // <-- додаємо ролi
    user.getRoles().add(manager);
    user = userRepository.save(user);

    // Перевiряємо: юзер має 2 ролi
    List<Role> roles = roleRepository.findByUsersId(user.getId());
    assertEquals(2, roles.size());

    // Зворотнiй зв'язок: роль -> юзери
    Role foundRole = roleRepository.findByName("ADMIN").orElseThrow();
    assertTrue(userRepository.findByRolesId(foundRole.getId()).stream()
            .anyMatch(u -> u.getSurname().equals("Хмельницький")));
}
```

## 4. FetchType: LAZY vs EAGER

Коли Hibernate завантажує сутнiсть з БД, вiн має вирiшити: чи вантажити пов'язанi сутностi одразу, чи потiм?

| FetchType | Поведiнка | Коли використовувати |
|-----------|-----------|---------------------|
| `LAZY` | Завантажує пов'язанi данi тiльки при зверненнi | За замовчуванням для колекцiй (`OneToMany`, `ManyToMany`) |
| `EAGER` | Завантажує одразу разом з основною сутнiстю | За замовчуванням для `ManyToOne`, `OneToOne` |

```java
// LAZY — банк НЕ завантажиться поки ми не викличемо user.getBank()
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "bank_id")
private Bank bank;
```

**Правило:** завжди використовуй `LAZY`. Якщо потрiбнi пов'язанi данi — завантажуй явно через запит або `@Transactional`.

### Проблема LazyInitializationException

```java
User user = userRepository.findById(1L).orElseThrow();
user.getBank().getName(); // LazyInitializationException!
```

Hibernate-сесiя вже закрита, а ми намагаємось довантажити банк. Рiшення:

1. Виконувати в `@Transactional` методi (сесiя залишається вiдкритою)
2. Використовувати спецiальний запит з JOIN FETCH
3. Використовувати окремий repository-метод (як ми робимо в проектi)

## 5. Cascade та orphanRemoval

### CascadeType

Каскаднi операцiї — це коли дiя над батькiвською сутнiстю автоматично поширюється на дочiрнi.

```java
@OneToMany(mappedBy = "bank", cascade = CascadeType.ALL, orphanRemoval = true)
private List<User> users = new ArrayList<>();
```

| Тип | Що робить |
|-----|-----------|
| `PERSIST` | Зберегти банк = зберегти i його юзерiв |
| `MERGE` | Оновити банк = оновити i юзерiв |
| `REMOVE` | Видалити банк = видалити i юзерiв |
| `ALL` | Все вище разом |

### orphanRemoval

```java
bank.getUsers().remove(user); // юзер стає "сиротою"
bankRepository.save(bank);     // Hibernate видалить його з БД
```

Без `orphanRemoval = true` юзер залишився б в БД з `bank_id = null`.

## 6. Транзакцiї (@Transactional)

### Що таке транзакцiя?

Транзакцiя — це група операцiй з БД, якi виконуються як **одне цiле**. Або всi операцiї успiшнi (commit), або жодна (rollback).

Класичний приклад: переказ грошей мiж рахунками.

```
1. Зняти 100 грн з рахунку A    ✓
2. Додати 100 грн на рахунок B  ✗ (помилка!)
```

Без транзакцiї: грошi зникли з A, але не з'явились на B. З транзакцiєю: обидвi операцiї вiдкотяться, грошi залишаться на A.

### ACID

Транзакцiї гарантують чотири властивостi:

| Властивiсть | Опис |
|-------------|------|
| **A**tomicity (Атомарнiсть) | Всi операцiї або виконуються, або жодна |
| **C**onsistency (Узгодженiсть) | БД переходить з одного валiдного стану в iнший |
| **I**solation (Iзольованiсть) | Паралельнi транзакцiї не бачать промiжнi стани одна одної |
| **D**urability (Стiйкiсть) | Пiсля commit данi збережено навiть при збої системи |

### @Transactional в Spring

Анотацiя `@Transactional` говорить Spring: "Обгорни цей метод в транзакцiю".

```java
@Service
@RequiredArgsConstructor
public class BankManagementServiceImpl implements BankManagementService {

    @Override
    @Transactional
    public List<UserDto> getUsersByBank(Long bankId) {
        final List<User> users = userRepository.findAllByBankId(bankId);
        return users.stream()
                .map(user -> userMapper.toDto(user))  // LAZY-зв'язки працюють в межах транзакцiї
                .toList();
    }

    @Override
    @Transactional
    public void deleteUser(Long userId) {
        userRepository.deleteById(userId);  // якщо тут впаде — нiчого не видалиться
    }
}
```

### Як це працює пiд капотом

```
Виклик методу з @Transactional
        |
        v
Spring створює проксi-об'єкт
        |
        v
BEGIN TRANSACTION
        |
        v
Виконується код методу
        |
    успiх?
   /       \
  так       нi (RuntimeException)
  |          |
COMMIT    ROLLBACK
```

### Приклад: успiшна реєстрацiя

```java
public User registerUserWithProfileAndRole(String name, String surname,
        String email, String phone, String roleName) {

    // 1. Створюємо юзера
    User user = new User();
    user.setName(name);
    user.setSurname(surname);
    user = userRepository.save(user);       // INSERT INTO users

    // 2. Створюємо профiль (OneToOne)
    UserProfile profile = new UserProfile();
    profile.setEmail(email);
    profile.setPhone(phone);
    profile.setUser(user);
    userProfileRepository.save(profile);     // INSERT INTO user_profile

    // 3. Знаходимо або створюємо роль (ManyToMany)
    Role role = roleRepository.findByName(roleName)
            .orElseGet(() -> {
                Role newRole = new Role();
                newRole.setName(roleName);
                return roleRepository.save(newRole);  // INSERT INTO role (якщо нова)
            });
    user.getRoles().add(role);

    return userRepository.save(user);        // INSERT INTO user_role
}
```

Три таблицi оновлюються в одному методi. Якщо будь-який крок впаде — все вiдкотиться.

### Приклад: rollback при помилцi

```java
@Transactional
public User registerUserThatFails(String name, String surname, String email) {
    User user = new User();
    user.setName(name);
    user.setSurname(surname);
    user = userRepository.save(user);           // INSERT — але ще не commit!

    UserProfile profile = new UserProfile();
    profile.setEmail(email);
    profile.setPhone("invalid");
    profile.setUser(user);
    userProfileRepository.save(profile);         // INSERT — але ще не commit!

    // Помилка! Все що вище — вiдкотиться
    throw new RuntimeException("Щось пiшло не так — транзакцiя вiдкотиться!");
}
```

### Тест на rollback

```java
@Test
void testTransactional_rollbackOnError() {
    long countBefore = userRepository.count();

    assertThrows(RuntimeException.class, () ->
            registrationService.registerUserThatFails("Fail", "User", "fail@test.com")
    );

    // Юзер НЕ збережений — транзакцiя вiдкотилась
    long countAfter = userRepository.count();
    assertEquals(countBefore, countAfter);  // кiлькiсть не змiнилась!
}
```

### Важливi нюанси @Transactional

**1. Працює тiльки при зовнiшньому виклику**

```java
@Service
public class MyService {

    public void methodA() {
        methodB();  // @Transactional НЕ працюватиме! Виклик всерединi того ж класу
    }

    @Transactional
    public void methodB() {
        // ...
    }
}
```

Spring створює проксi навколо бiна. Коли ви викликаєте метод з того ж класу — проксi не задiяний, транзакцiя не вiдкривається.

**2. За замовчуванням rollback тiльки на RuntimeException**

```java
@Transactional  // checked exception НЕ викличе rollback
public void doWork() throws IOException {
    // ...
    throw new IOException("oops");  // транзакцiя закомiтиться!
}

@Transactional(rollbackOn = Exception.class)  // тепер rollback на будь-який exception
public void doWorkSafe() throws IOException {
    // ...
}
```

**3. readOnly для запитiв на читання**

```java
@Transactional(readOnly = true)
public List<UserDto> getAllUsers() {
    // Hibernate не буде робити dirty-checking — працює швидше
    return userRepository.findAll().stream()
            .map(userMapper::toDto)
            .toList();
}
```

## 8. Пiдсумок

| Концепцiя | Що запам'ятати |
|-----------|---------------|
| `@OneToMany` / `@ManyToOne` | FK на сторонi "Many". `mappedBy` на сторонi "One" |
| `@OneToOne` | FK з UNIQUE constraint. `mappedBy` на зворотнiй сторонi |
| `@ManyToMany` | Потрiбна промiжна таблиця (`@JoinTable`). Використовуй `Set` |
| `FetchType.LAZY` | Завжди LAZY. Завантажуй данi явно через запити або `@Transactional` |
| `CascadeType.ALL` | Каскаднi операцiї на дочiрнi сутностi. Обережно з `REMOVE` |
| `@Transactional` | Всi операцiї — або commit, або rollback. Працює тiльки через проксi |
| Rollback | RuntimeException = rollback. Checked exception = commit (за замовчуванням) |
