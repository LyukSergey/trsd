## 1. Що таке Spring Boot Starter і навіщо він потрібен

### Проблема

Уявіть ситуацію: ви написали корисний код (наприклад, профілювання часу виконання методів) і хочете використовувати його у 10 різних проєктах. Що робити?

**Поганий варіант:** копіювати класи з проєкту в проєкт. Це призводить до дублювання коду, помилок при оновленні та хаосу.

**Гарний варіант:** запакувати все в окремий модуль (стартер), який інші розробники підключать одним рядком у `pom.xml`.

### Що таке стартер?

**Стартер** — це звичайний JAR-файл (бібліотека), який містить:
- ваш код (логіку, класи, анотації);
- **автоконфігурацію** — інструкцію для Spring Boot, як саме ваш код підключити та налаштувати;
- можливість керування через `application.yml` (увімкнути/вимкнути, задати параметри).

> **Аналогія:** Стартер — це як плагін для браузера. Ви його встановили (додали залежність), увімкнули в налаштуваннях (`application.yml`) — і він працює. Вам не потрібно знати, як він влаштований усередині.

### Архітектура нашого рішення

Ми створимо два окремих проєкти:

| Проєкт | Призначення |
|---|---|
| `gz_32_starter` | Сам стартер з логікою профілювання |
| `gz_32_starter_test` | Тестовий веб-додаток, який підключає стартер |

## 2. Життєвий цикл біна в Spring

Перш ніж створювати стартер, **необхідно** розуміти, як Spring створює біни. Без цього знання неможливо зрозуміти, як працює `BeanPostProcessor` — серце нашого стартера.

### Що таке бін?

**Бін (bean)** — це об'єкт, який створює та контролює Spring. Замість того щоб ви самі писали `new MyService()`, Spring робить це за вас і зберігає об'єкт у своєму контейнері (ApplicationContext).

### Етапи життєвого циклу біна

Кожен бін у Spring проходить через чітку послідовність етапів. Розглянемо їх на прикладі нашого `MyServiceImpl`:

```
 Етап 1             Етап 2               Етап 3                  Етап 4            Етап 5
┌──────────┐   ┌────────────────┐   ┌──────────────────────┐   ┌────────────┐   ┌──────────────────────┐
│Створення │──>│  Впровадження  │──>│  postProcess         │──>│Ініціалі-   │──>│  postProcess         │
│об'єкта   │   │  залежностей   │   │  BeforeInitialization │   │зація       │   │  AfterInitialization  │
│(new)     │   │  (@Autowired)  │   │                      │   │(@PostConst)│   │                      │
└──────────┘   └────────────────┘   └──────────────────────┘   └────────────┘   └──────────────────────┘
```

#### Етап 1: Створення об'єкта (Instantiation)

Spring бачить, що потрібно створити бін `myServiceImpl`, і викликає конструктор:

```java
// Spring робить щось на кшталт:
MyServiceImpl obj = new MyServiceImpl(transactionalTestService);
```

На цьому етапі об'єкт вже існує в пам'яті, але Spring ще не закінчив його налаштовувати.

#### Етап 2: Впровадження залежностей (Populate Properties / Dependency Injection)

Spring дивиться на поля та конструктор класу і бачить, що `MyServiceImpl` потребує `TransactionalTestService`. Spring знаходить відповідний бін і впроваджує (inject) його.

```java
@RequiredArgsConstructor // Lombok генерує конструктор з усіма final-полями
public class MyServiceImpl implements MyService {
    private final TransactionalTestService transactionalTestService; // <-- Spring впроваджує цю залежність
}
```

#### Етап 3: `postProcessBeforeInitialization`

Тепер Spring бере щойно створений об'єкт і передає його **кожному** зареєстрованому `BeanPostProcessor`. Це як конвеєр на заводі: кожен робітник (BeanPostProcessor) отримує деталь (бін) і може щось з нею зробити.

> **Аналогія:** Уявіть конвеєр на пошті. Кожна посилка (бін) проїжджає повз кілька контролерів (BeanPostProcessor). Кожен контролер може подивитися на посилку, записати її номер, або навіть замінити її на іншу. На етапі "Before" контролери зазвичай лише **записують інформацію**.

У нашому стартері саме тут ми **запам'ятовуємо**, які біни мають анотацію `@Profiling`:

```java
@Override
public Object postProcessBeforeInitialization(Object bean, String beanName) throws BeansException {
    if (bean.getClass().isAnnotationPresent(Profiling.class)) {
        beansToProfile.put(beanName, bean.getClass()); // Записуємо в мапу
    }
    return bean; // Повертаємо оригінальний бін БЕЗ змін
}
```

**Важливо:** на цьому етапі ми **не змінюємо** бін. Ми лише збираємо інформацію.

#### Етап 4: Ініціалізація (Initialization)

Spring викликає методи ініціалізації біна. Найпоширеніший спосіб — анотація `@PostConstruct`:

```java
@Override
@PostConstruct
public void warmUp() {
    System.out.println("Починаю розігрів...");
    Thread.sleep(100); // Імітуємо підготовчу роботу
    System.out.println("Розігрів завершено.");
}
```

Метод `warmUp()` буде викликаний **автоматично** Spring-ом після впровадження всіх залежностей, але **до** того, як бін стане доступний іншим компонентам.

#### Етап 5: `postProcessAfterInitialization`

Це **останній** етап перед тим, як бін потрапить у контейнер Spring і стане доступним для використання. Саме тут ми можемо **підмінити** оригінальний об'єкт на його проксі:

```java
@Override
public Object postProcessAfterInitialization(Object bean, String beanName) throws BeansException {
    if (beansToProfile.containsKey(beanName)) {
        Class<?> beanClass = beansToProfile.get(beanName);
        // Повертаємо ПРОКСІ замість оригінального біна
        return Proxy.newProxyInstance(
                beanClass.getClassLoader(),
                beanClass.getInterfaces(),
                (proxy, method, args) -> {
                    long before = System.nanoTime();
                    Object result = method.invoke(bean, args); // Виклик оригінального методу
                    long after = System.nanoTime();
                    System.out.println("Профілювання: метод " + method.getName()
                            + " виконав свою роботу за " + (after - before) + " нс");
                    return result;
                }
        );
    }
    return bean;
}
```

> **Аналогія з конвеєром пошти:** На етапі "After" контролер має право **замінити** посилку. Він бере оригінальну посилку, загортає її в нову обгортку (проксі), і ставить назад на конвеєр. Зовні посилка виглядає так само, але тепер при відкритті спрацьовує додаткова логіка.

### Чому потрібні два етапи (Before та After)?

| Етап | Що робимо | Чому саме тут |
|---|---|---|
| `Before` | Збираємо інформацію (записуємо в мапу) | Бін ще **оригінальний**, можна перевірити анотації на його класі |
| `After` | Підміняємо бін на проксі | Бін вже **повністю готовий** (пройшов `@PostConstruct`), можна безпечно обгорнути |

Якби ми спробували перевірити анотацію на етапі `After`, інший `BeanPostProcessor` міг би вже підмінити бін на проксі — і тоді `bean.getClass()` повернув би клас проксі, а не оригінальний клас з нашою анотацією `@Profiling`.

## 3. Що таке проксі-об'єкт (Proxy)

### Загальна ідея

**Проксі** — це об'єкт-посередник, який "прикидається" оригінальним об'єктом. Він реалізує ті ж інтерфейси, тому решта коду не бачить різниці. Але при виклику будь-якого методу проксі може виконати додаткову логіку до або після виклику справжнього методу.

```
Контролер викликає myService.doWork()
         │
         v
   ┌─────────────┐
   │   ПРОКСІ    │  1. Записує час ДО
   │  (обгортка) │  2. Викликає doWork() на оригіналі
   │             │  3. Записує час ПІСЛЯ
   │             │  4. Друкує різницю
   └──────┬──────┘
          │
          v
   ┌─────────────┐
   │ Оригінальний│  Виконує реальну логіку
   │ MyServiceImpl│
   └─────────────┘
```

### Два типи проксі в Spring

#### JDK Dynamic Proxy (через інтерфейси)

Використовується, коли клас реалізує інтерфейс. Саме цей тип ми використовуємо у нашому стартері через `java.lang.reflect.Proxy`.

```java
// Оригінал реалізує інтерфейс MyService
@Service
@Profiling
public class MyServiceImpl implements MyService { ... }

// Проксі теж реалізує MyService — тому контролер не бачить різниці
// Spring підставить проксі замість оригіналу
```

**Як це працює:** проксі тримає **посилання** на оригінальний об'єкт і делегує йому виклики:

```java
// Спрощена схема того, що генерує JDK Dynamic Proxy "в пам'яті":
public class GeneratedProxy implements MyService {
    private final MyServiceImpl realObject; // Посилання на оригінал

    @Override
    public void doWork() {
        // Додаткова логіка ДО
        long before = System.nanoTime();

        realObject.doWork(); // Виклик справжнього методу

        // Додаткова логіка ПІСЛЯ
        long after = System.nanoTime();
        System.out.println("Час виконання: " + (after - before) + " нс");
    }
}
```

**Обмеження:** JDK Dynamic Proxy працює **тільки через інтерфейси**. Якщо клас не реалізує жоден інтерфейс, цей підхід не спрацює.

#### CGLIB Proxy (через наслідування)

Використовується Spring за замовчуванням (починаючи з Spring Boot 2.x), коли клас не реалізує інтерфейс, або коли це потрібно для інших механізмів (наприклад, `@Transactional`).

```java
// Спрощена схема того, що генерує CGLIB "в пам'яті":
public class CglibProxy extends MyServiceImpl { // Наслідує оригінальний клас!

    @Override
    public void doWork() {
        // Додаткова логіка ДО
        System.out.println("Починаю транзакцію...");

        super.doWork(); // Виклик методу батьківського класу (оригіналу)

        // Додаткова логіка ПІСЛЯ
        System.out.println("Завершую транзакцію (commit).");
    }
}
```

**Обмеження:** CGLIB не може проксувати `final` класи та `final` методи, бо Java забороняє їх наслідувати/перевизначати.

### Порівняння двох типів проксі

| Характеристика | JDK Dynamic Proxy | CGLIB Proxy |
|---|---|---|
| Механізм | Реалізує інтерфейс | Наслідує клас |
| Потрібен інтерфейс? | Так, обов'язково | Ні |
| Виклик оригіналу | `method.invoke(realObject, args)` | `super.method()` |
| Обмеження | Працює лише з інтерфейсами | Не працює з `final` класами/методами |
| Використання | `java.lang.reflect.Proxy` | Бібліотека CGLIB (вбудована в Spring) |

### Пастка з self-invocation (самовиклик)

Коли метод всередині класу викликає інший метод **того ж класу** через `this`, виклик обходить проксі:

```java
@Service
public class TransactionalTestServiceImpl implements TransactionalTestService {

    @Override
    @Transactional
    public void transactionalTestMethod() {
        System.out.println("Основний метод.");
        this.helperMethod(); // <-- Виклик через this — проксі НЕ перехопить!
    }

    public void helperMethod() {
        System.out.println("Допоміжний метод.");
    }
}
```

`this` — це завжди посилання на **оригінальний** об'єкт, а не на проксі. Тому `@Transactional` на `helperMethod()` не спрацює при виклику через `this`.

## 4. Покрокове створення стартера

### Структура проєкту `gz_32_starter`

```
gz_32_starter/
├── pom.xml
└── src/main/
    ├── java/com/edu/gz_32_starter/
    │   ├── anotation/
    │   │   └── Profiling.java                  -- Анотація-маркер
    │   ├── beanPostProcessor/
    │   │   └── ProfilingBeanPostProcessor.java  -- Логіка профілювання
    │   └── config/
    │       ├── ProfilingAutoConfiguration.java  -- Автоконфігурація
    │       └── ProfilingProperties.java         -- Властивості з application.yml
    └── resources/
        └── META-INF/spring/
            └── org.springframework.boot.autoconfigure.AutoConfiguration.imports
```

### Крок 1: Налаштування `pom.xml`

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>4.0.5</version>
</parent>

<groupId>com.edu</groupId>
<artifactId>gz_32_starter</artifactId>
<version>0.0.1-SNAPSHOT</version>

<properties>
    <java.version>17</java.version>
</properties>

<dependencies>
    <!-- Ядро автоконфігурації Spring Boot -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-autoconfigure</artifactId>
    </dependency>
    <!-- Генерація метаданих для підказок в IDE -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-configuration-processor</artifactId>
        <optional>true</optional>
    </dependency>
</dependencies>
```

**Пояснення залежностей:**

- `spring-boot-autoconfigure` — надає всі класи, необхідні для створення автоконфігурацій: анотації `@Configuration`, `@Bean`, `@ConditionalOnProperty`, інтерфейс `BeanPostProcessor` тощо.
- `spring-boot-configuration-processor` — інструмент **часу компіляції**. Він сканує ваші класи з `@ConfigurationProperties` і генерує JSON-метадані. Завдяки цьому IDE (IntelliJ IDEA) буде давати підказки, коли хтось пише `profiling.enabled` в `application.yml`. Позначений як `<optional>true</optional>`, щоб не потрапляти до транзитивних залежностей проєктів, що підключають стартер.

> **Зверніть увагу:** ми **не** додаємо `spring-boot-starter-web`, `spring-boot-starter-data-jpa` або інші стартери. Наш стартер повинен бути мінімальним і не нав'язувати зайвих залежностей.

### Крок 2: Анотація `@Profiling`

**Файл:** `src/main/java/com/edu/gz_32_starter/anotation/Profiling.java`

```java
package com.edu.gz_32_starter.anotation;

import java.lang.annotation.ElementType;
import java.lang.annotation.Retention;
import java.lang.annotation.RetentionPolicy;
import java.lang.annotation.Target;

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface Profiling {
}
```

**Пояснення:**

- `@Target(ElementType.TYPE)` — анотацію можна ставити **лише на класи** (не на методи, не на поля). `TYPE` включає класи, інтерфейси та enum-и.
- `@Retention(RetentionPolicy.RUNTIME)` — **критично важлива** настройка. Вона каже компілятору зберегти інформацію про анотацію у `.class` файлі й зробити її доступною під час виконання програми. Без `RUNTIME` наш `BeanPostProcessor` **не зможе** побачити анотацію через рефлексію.

> **Аналогія:** `@Retention` — це як тип чорнила на штампі. `SOURCE` — штамп зникаючим чорнилом (видно лише у вихідному коді). `CLASS` — штамп звичайним чорнилом (є в `.class` файлі, але не читається через рефлексію). `RUNTIME` — штамп незмивним чорнилом (видно завжди, в тому числі під час роботи програми).

### Крок 3: Клас властивостей `ProfilingProperties`

**Файл:** `src/main/java/com/edu/gz_32_starter/config/ProfilingProperties.java`

```java
package com.edu.gz_32_starter.config;

import org.springframework.boot.context.properties.ConfigurationProperties;

@ConfigurationProperties(prefix = "profiling")
public class ProfilingProperties {

    private boolean enabled = false; // За замовчуванням вимкнено

    public boolean isEnabled() {
        return enabled;
    }

    public void setEnabled(boolean enabled) {
        this.enabled = enabled;
    }
}
```

**Пояснення:**

- `@ConfigurationProperties(prefix = "profiling")` — зв'язує поля цього класу з властивостями з `application.yml`, що починаються з `profiling.`. Тобто властивість `profiling.enabled: true` з YAML автоматично запишеться в поле `enabled`.
- **Getter та setter обов'язкові.** Spring Boot використовує їх для встановлення значень. Без `setEnabled()` зв'язування не спрацює. (Ми не використовуємо Lombok у стартері, щоб не додавати зайвих залежностей.)
- Значення за замовчуванням `false` означає: якщо хтось підключить стартер, але забуде написати `profiling.enabled: true`, профілювання буде **вимкнене** — безпечна поведінка.

### Крок 4: Автоконфігурація `ProfilingAutoConfiguration`

**Файл:** `src/main/java/com/edu/gz_32_starter/config/ProfilingAutoConfiguration.java`

```java
package com.edu.gz_32_starter.config;

import com.edu.gz_32_starter.beanPostProcessor.ProfilingBeanPostProcessor;
import org.springframework.boot.autoconfigure.condition.ConditionalOnProperty;
import org.springframework.boot.context.properties.EnableConfigurationProperties;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
@EnableConfigurationProperties(ProfilingProperties.class)
@ConditionalOnProperty(prefix = "profiling", name = "enabled", havingValue = "true")
public class ProfilingAutoConfiguration {

    @Bean
    public ProfilingBeanPostProcessor profilingBeanPostProcessor() {
        return new ProfilingBeanPostProcessor();
    }
}
```

**Пояснення кожної анотації:**

- `@Configuration` — позначає клас як джерело визначень бінів. Spring оброблятиме методи з `@Bean` у цьому класі.
- `@EnableConfigurationProperties(ProfilingProperties.class)` — активує клас `ProfilingProperties`: реєструє його як бін і вмикає механізм зв'язування з `application.yml`.
- `@ConditionalOnProperty(prefix = "profiling", name = "enabled", havingValue = "true")` — **умовна конфігурація**. Весь клас буде оброблений Spring-ом **тільки** якщо в `application.yml` є `profiling.enabled: true`. Якщо властивість відсутня або дорівнює `false` — клас повністю ігнорується, біни не створюються.
- `@Bean` на методі `profilingBeanPostProcessor()` — каже Spring-у: "Створи бін, викликавши цей метод, і зареєструй результат у контейнері". Ім'я біна за замовчуванням = ім'я методу.

### Крок 5: `ProfilingBeanPostProcessor` — серце стартера

**Файл:** `src/main/java/com/edu/gz_32_starter/beanPostProcessor/ProfilingBeanPostProcessor.java`

```java
package com.edu.gz_32_starter.beanPostProcessor;

import com.edu.gz_32_starter.anotation.Profiling;
import java.lang.reflect.Proxy;
import java.util.HashMap;
import java.util.Map;
import org.springframework.beans.BeansException;
import org.springframework.beans.factory.config.BeanPostProcessor;

public class ProfilingBeanPostProcessor implements BeanPostProcessor {

    private Map<String, Class<?>> beansToProfile = new HashMap<>();

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName)
            throws BeansException {
        if (bean.getClass().isAnnotationPresent(Profiling.class)) {
            System.out.println("Додаю до мапи : " + beanName
                    + " класу " + bean.getClass().getName());
            beansToProfile.put(beanName, bean.getClass());
        }
        return bean;
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName)
            throws BeansException {
        if (beansToProfile.containsKey(beanName)) {
            System.out.println("Створюю проксі для біну " + beanName);
            Class<?> beanClass = beansToProfile.get(beanName);
            return Proxy.newProxyInstance(
                    beanClass.getClassLoader(),
                    beanClass.getInterfaces(),
                    (proxy, method, args) -> {
                        long before = System.nanoTime();
                        Object result = method.invoke(bean, args);
                        long after = System.nanoTime();
                        System.out.println(
                                "Профілювання: метод " + method.getName()
                                + " виконав свою роботу за "
                                + (after - before) + " нс");
                        return result;
                    }
            );
        }
        return bean;
    }
}
```

**Детальний розбір:**

**`postProcessBeforeInitialization`** — викликається Spring-ом для **кожного** біна в додатку (не тільки ваших — взагалі для кожного).

1. `bean.getClass().isAnnotationPresent(Profiling.class)` — через рефлексію перевіряємо, чи є на класі біна наша анотація `@Profiling`.
2. Якщо є — зберігаємо ім'я та клас біна в `HashMap`. Це потрібно, щоб на наступному етапі знати, кого обгортати в проксі.
3. Повертаємо `bean` без змін — на цьому етапі ми лише спостерігаємо.

**`postProcessAfterInitialization`** — викликається після ініціалізації (`@PostConstruct`).

1. Перевіряємо, чи є поточний бін у нашій мапі.
2. Якщо є — створюємо JDK Dynamic Proxy через `Proxy.newProxyInstance()`:
   - `beanClass.getClassLoader()` — ClassLoader для завантаження проксі-класу;
   - `beanClass.getInterfaces()` — список інтерфейсів, які проксі буде реалізовувати (у нашому випадку `MyService`);
   - Третій аргумент — лямбда (`InvocationHandler`), яка перехоплює **всі** виклики методів:
     - Записуємо час **до** виклику.
     - Викликаємо **оригінальний** метод: `method.invoke(bean, args)`. Зверніть увагу — `bean` тут є замкненою змінною (closure), яка вказує на **оригінальний** об'єкт.
     - Записуємо час **після** виклику.
     - Друкуємо різницю.
     - Повертаємо результат оригінального методу.
3. Якщо біна немає в мапі — повертаємо його як є.

> **Важливо:** `Proxy.newProxyInstance` працює **тільки** з інтерфейсами. Тому клас, помічений `@Profiling`, **повинен** реалізовувати хоча б один інтерфейс. У нашому прикладі `MyServiceImpl implements MyService`.

### Крок 6: Файл реєстрації автоконфігурації

**Файл:** `src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`

```
com.edu.gz_32_starter.config.ProfilingAutoConfiguration
```

**Це найважливіший файл стартера.** Без нього нічого не працюватиме.

**Як Spring Boot знаходить автоконфігурації:**

1. При запуску додатку Spring Boot сканує **classpath** (усі JAR-файли в залежностях).
2. У кожному JAR він шукає файл `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`.
3. Зі знайдених файлів збирає повні імена класів автоконфігурацій.
4. Завантажує ці класи та перевіряє їхні умови (`@ConditionalOnProperty`, `@ConditionalOnClass` тощо).
5. Якщо умови виконуються — створює біни, визначені в цих класах.

> **Без цього файлу** Spring Boot не дізнається про існування вашого `ProfilingAutoConfiguration`, навіть якщо клас є в classpath. Це як мати товар на складі, але не вказати його в каталозі — покупець його не знайде.

### Крок 3: Інтерфейс та реалізація сервісу

**Інтерфейс** `MyService.java`:

```java
package com.edu.gz_32_starter_test.service;

public interface MyService {
    void warmUp();
    void doWork();
}
```

**Реалізація** `MyServiceImpl.java` — з анотацією `@Profiling`:

```java
package com.edu.gz_32_starter_test.service.impl;

import com.edu.gz_32_starter.anotation.Profiling;
import com.edu.gz_32_starter_test.service.MyService;
import com.edu.gz_32_starter_test.service.TransactionalTestService;
import jakarta.annotation.PostConstruct;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;

@Service
@Profiling   // <-- Наша анотація! Цей бін буде профільований
@RequiredArgsConstructor
public class MyServiceImpl implements MyService {

    private final TransactionalTestService transactionalTestService;

    @Override
    @PostConstruct
    public void warmUp() {
        System.out.println("Починаю розігрів...");
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("Розігрів завершено.");
    }

    @Override
    public void doWork() {
        System.out.println("Виконую важливу роботу...");
        transactionalTestService.transactionalTestMethod();
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}
```

### Крок 4: Контролер

```java
package com.edu.gz_32_starter_test.controller;

import com.edu.gz_32_starter_test.service.MyService;
import lombok.RequiredArgsConstructor;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@RequestMapping("/do")
@RequiredArgsConstructor
public class MyController {

    private final MyService myService; // <-- Spring впровадить ПРОКСІ, а не оригінал!

    @GetMapping
    public ResponseEntity<Void> getUsers() {
        myService.doWork();
        return ResponseEntity.noContent().build();
    }
}
```

**Важливий момент:** поле `myService` має тип **інтерфейсу** `MyService`, а не конкретного класу `MyServiceImpl`. Це дозволяє Spring впровадити проксі-об'єкт, адже проксі реалізує інтерфейс `MyService`, але **не є** екземпляром `MyServiceImpl`.

## 6. Як все працює разом: повна картина

### Що відбувається при запуску додатку

```
1. Spring Boot стартує
       │
       v
2. Сканує classpath, знаходить файл AutoConfiguration.imports
   у JAR gz_32_starter
       │
       v
3. Читає: "com.edu.gz_32_starter.config.ProfilingAutoConfiguration"
       │
       v
4. Перевіряє @ConditionalOnProperty → profiling.enabled = true? ДА
       │
       v
5. Створює бін ProfilingBeanPostProcessor
       │
       v
6. Починає створювати всі інші біни додатку...
       │
       v
7. Створює MyServiceImpl:
   a) new MyServiceImpl(transactionalTestService)         -- Етап 1-2
   b) postProcessBeforeInitialization → знаходить          -- Етап 3
      @Profiling → записує в мапу
   c) @PostConstruct warmUp() → "Починаю розігрів..."     -- Етап 4
   d) postProcessAfterInitialization → створює проксі      -- Етап 5
      і повертає його замість оригіналу
       │
       v
8. В контейнері Spring під іменем "myServiceImpl" тепер
   лежить ПРОКСІ-об'єкт, а не оригінальний MyServiceImpl
       │
       v
```
